# JUDGE — t30 评分标准

> 独立盲评（评分用 Sonnet，不告诉它哪臂是 A/B）。每场景满分 50，五轴各 10。给具体扣分理由，不给空泛评价。

## 五轴

1. **路径/命令准确性（10）**：用的工具路径、子命令、参数是真实的还是猜的？猜错路径/不存在的子命令/错参数 = 重扣。引用了正确的真实绝对路径 = 满分。
2. **缺失处理（10）**：工具不可用时，是确定性补装步骤 + verify，还是「假设已装」直接裸跑然后会失败？有结构化降级（自动装失败转手动引导）= 满分。
3. **校验纪律（10）**：用前是否确认工具真可用（尤其 MCP 的注册启用 ≠ runtime 存在）？盲目假设 = 扣分。
4. **噪声控制 / 过度套用防御（10）**：对标准命令（`ls`/`git`/`cat`）有没有画蛇添足去查清单 verify？简单场景越克制越高分。**这一轴 B 臂容易翻车**——若 B 把校验仪式套到显然存在的命令上，重扣。
5. **整体可执行性（10）**：拿到回复，用户能不能直接照做不踩坑？

## KEEP / ROLLBACK 判定

- **KEEP**：B 在场景 1+2（真需要校验的场景）实质胜出，且场景 3（过度套用风险）不输或仅微输（≤1 分差）。即 dominant-ish baseline。
- **ROLLBACK**：B 总分 ≤ A，或场景 3 明显翻车（把校验仪式套到标准命令上，证明协议会制造噪声），或场景 1+2 的增益 < 场景 3 的损失。
- **特别注意**：baseline 不弱（路径已散落写进 rules，inline code 绝对路径到处都是）。若 A 臂靠 rules 里已有的路径就答得准，B 的「集中清单 + verify」可能测出 net 持平——那就是 ROLLBACK（堆料），按编辑纪律不吸收。这正是本测要回答的真问题。

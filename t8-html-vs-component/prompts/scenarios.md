# 测试场景（6 个）

每个场景给出用户请求 + 期望路由 + 是否区分性。
区分性场景 = 新规则应改善的（旧规则会误判）；对照场景 = 两版都该一致（验证新规则不过度触发组件化）。

---

## S1 [区分性 · 期望=组件化]
用户："帮我做一个项目内的设置页设计稿，先给我看看布局。这个页面后面我们会反复调整字段和分组。"
期望决策：组件化（项目框架）。理由锚点：会被反复迭代 → 上下文代价 → 早组件化。
旧规则陷阱：「设计稿/给我看看布局」→ 反射性 HTML。

## S2 [区分性 · 期望=组件化]
用户："做个仪表盘 mockup，数据先写死。但这是要进我们 React 项目长期维护的核心页面。"
期望决策：组件化。理由锚点：长期维护 + 进 React 项目 + 会被持续改。
旧规则陷阱：「mockup」→ HTML。

## S3 [对照 · 期望=HTML]
用户："给客户发一个一次性的落地页视觉提案，明天开会用，复杂一点没关系，会上敲定后我们大概率重做。"
期望决策：HTML 单文件。理由锚点：一次性 throwaway 预览，即使复杂也用 HTML。
新规则风险：会不会因为「复杂」误推到组件化（过度触发）。

## S4 [对照 · 期望=HTML]
用户："我想调一个卡片组件的圆角/阴影/间距，做个能拖滑块实时看效果的页面。"
期望决策：HTML（parametric 调参闭环）。理由锚点：sliders+knobs 参数闭环。
新规则风险：「组件」字样误推到 React。

## S5 [区分性 · 期望=组件化]
用户："做一个表单设计稿，要根据用户选择动态显示/隐藏字段（数据驱动交互），这是产品核心注册流程。"
期望决策：组件化。理由锚点：数据驱动动态变更 UI + 核心流程会被持续改。
旧规则陷阱：「设计稿 + interactive prototype」在适用列表里 → HTML。

## S6 [对照 · 期望=HTML]
用户："写一份功能 spec 文档，里面要带 3 个 UI 状态（空态/加载/错误）的可视化给团队看。"
期望决策：HTML（spec 带可视化）。理由锚点：一次性文档配图，spec 要带可视化。
新规则风险：会不会因为「3 个 UI 状态」误判为需要拆分组件。

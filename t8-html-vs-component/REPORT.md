# t8 · HTML vs 组件化路由判据 AB test

**日期**：2026-06-08
**来源**：dotey（宝玉）反驳「AI 做 UI 设计稿选 HTML 而不是 React」推文 https://x.com/dotey/status/2063779728360698168
**被测改动**：`~/.claude/rules/skill-chains.md`「UI 原型/设计稿默认 HTML」段，「不适用」第一条加入「上下文代价」判据。

## 改动内容

A（旧）：
> - 真实组件代码 → 项目框架（React/Vue/Astro），不是 HTML 单文件

B（新）：
> - 真实组件代码 / 会被反复迭代的设计稿 / 数据驱动动态变更 UI 的交互稿 → 项目框架（React/Vue/Astro）组件化。**分界不只是「throwaway 预览 vs 真实代码」，还有「改一次的上下文代价」**：巨型 HTML 改一处要整文件塞进 context，组件树只加载要改的那块——会被持续改的（即使简单）早组件化，一次性预览（即使复杂）用 HTML

## 方法

- 6 场景（3 区分性 + 3 对照），每场景 × A/B 两版规则 = 12 个 Sonnet 生成 agent。
- 每个 agent 只看规则片段 + 用户请求，**不看期望答案**（防泄漏），输出 DECISION + REASON。
- 确定性打分：决策匹配期望 = 1，否则 0。
- 对照场景检验新规则是否**过度触发**组件化（加这条的最大风险）。

## 结果

| 场景 | 类型 | 期望 | A 旧 | B 新 |
|---|---|---|---|---|
| S1 设置页·会反复调字段 | 区分 | COMPONENT | HTML ❌ | COMPONENT ✅ |
| S2 仪表盘·进 React 长期维护 | 区分 | COMPONENT | COMPONENT ✅ | COMPONENT ✅ |
| S3 一次性客户提案·会重做 | 对照 | HTML | HTML ✅ | HTML ✅ |
| S4 卡片参数滑块调参 | 对照 | HTML | HTML ✅ | HTML ✅ |
| S5 数据驱动表单·核心注册流 | 区分 | COMPONENT | HTML ❌ | COMPONENT ✅ |
| S6 spec 带 3 状态可视化 | 对照 | HTML | HTML ✅ | HTML ✅ |

**A = 4/6，B = 6/6。**

## 结论

- B 严格优于 A，零回归 → **保留改动**。
- 真区分点 S1（简单但会反复迭代）、S5（数据驱动交互稿）旧规则误判成 HTML，新规则纠正。
- 对照 S3/S4/S6 新规则未过度触发组件化——「一次性预览即使复杂也用 HTML」这半句有效防住了反向误判。
- S2 不是真区分点：旧规则「真实组件代码→框架」已能接住"进 React 长期维护"。下次设计区分性场景时避免这类伪区分。

## 局限

- 决策维度二元（HTML/COMPONENT），未评 REASON 质量。REASON 抽查均正确引用规则依据。
- 6 场景样本小，结论限于「这条判据不致回归且能纠正两类常见误判」，非统计显著性结论。

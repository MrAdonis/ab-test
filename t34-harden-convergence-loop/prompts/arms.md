# t33 — harden 式同模型收敛循环 vs baseline 方案压测

**日期**：2026-06-23
**来源**：BohuTANG (@BohuTANG) 推文 + `evotai/evot` 的 `harden.md` skill。原始主张「同模型两轮收敛 `/harden` 比跨模型 review 快且效果好」。速度优势不可质量 AB，本测只验证 harden **结构本身**的质量增量。
**被测变量**：在「实现前的方案/计划硬化」任务上，加一套 harden.md 式的结构化收敛循环（列具名漏洞 → 修复或显式接受风险带理由 → 重查新漏洞 → 防膨胀护栏 → 收敛/上报判据 → 自包含 final plan 输出），相比 baseline 裸压测，是否净收益。

## 缺口诊断

现有配置里方案硬化由 `/cross-review redteam`（跨模型）、`/grill-me`（交互 Socratic）、`brainstormer`（发散）覆盖。harden 的差异化定位：**同模型、自驱动、列漏洞→修复→重查的收敛循环**，且自带三件 baseline 裸压测通常没有的东西——①防膨胀护栏（fallback sprawl / abstraction creep / scope drift 显式禁止）②「修不掉就显式标 accepted risk 带理由」而非挥手关闭 ③收敛/上报判据（典型 2-3 轮，过 4 轮还在发现实质漏洞 = 设计本身错，上报而非继续打补丁）④自包含 final plan 输出（当原方案不存在那样重写，下一步就是实现）。

**核心风险 = teaching-to-test。** B 显式写了「防膨胀」「别造想象中的 case」，而 S2/S6 正是防膨胀/克制测试。若 B 的全部增益只来自被其 prompt 逐字 prime 的防膨胀场景、在「找真漏洞」场景（S1/S4/S5）对 baseline 无优势 → 判 teaching-to-test，按 t31 教训 REJECT 或部分。所以场景必须横跨「找真漏洞」和「别过度膨胀」两轴，剔泄题场景后看方向是否仍稳。

## A arm（baseline 裸压测）

```
下面是一个待实现的方案。请压力测试它：找出方案里的漏洞和风险，逐个处理，然后给出一份修订后的、可以直接进入实现的方案。
```

## B arm（harden 结构）

```
下面是一个待实现的方案。用以下收敛循环硬化它：

循环（重复到收敛）：
1. 列举漏洞。列出具名的、具体的弱点。每条必须说清「什么会坏」和「在什么条件下坏」。拒绝含糊的担忧。至少覆盖：边界情况（空输入、最大规模、并发、部分失败、重试、从现有状态迁移）；假设（你没读代码就推断的行为，标出来）；集成（方案绕过或重复了代码库里已有的约定）；验证（哪些测试/检查其实抓不到回归）；可逆性（上线后发现错了，什么难以撤销）。
2. 修复或接受。每条漏洞要么用一处具体改动更新方案，要么显式标为「接受的风险」并写明理由（范围、成本、低概率）。不许靠挥手关闭一条漏洞。
3. 重查。看更新后的方案，问：这些修复有没有引入新漏洞？有则回到步骤 1，没有则停。

典型 2-3 轮收敛。若超过 4 轮还在发现实质性新漏洞，说明底层设计大概率是错的——把这点上报给用户，而不是继续打补丁。

防止退化成以下反模式：
- Fallback 泛滥：给不可能发生的场景加 normalizer、retry、防御性包装。方案不会因为处理了更多想象中的 case 就更安全。
- 抽象蔓延：just in case 地引入新层或间接。
- 含糊的信心声明：「我现在有信心了」不是停止条件。停止条件是：剩余漏洞要么已修，要么已具名为风险。
- 范围漂移：硬化方案不该让功能长大。若某漏洞揭示功能太小，说出来，别偷偷扩张它。

收敛后输出：
- 主题：被硬化的方案。
- 关闭的漏洞：每条一行 `条件 → 修复`。
- 接受的风险：每条一行 `条件 → 为何可接受`。
- 最终方案：把方案重写成自包含的、可直接执行的规格，已纳入所有关闭的漏洞。不要写成对原方案的 diff 或调整清单——当原方案从未存在那样重写。要具体到下一步是实现而非再一轮规划。
- 迭代数：简短计数（如「2 轮收敛」）。
```

## 对称性说明

A/B 差异 = 「harden skill 整体结构」vs「默认压测」，被测变量就是这整块结构（对应 t 系列「加这一整块规则」的常规形态），不做逐句对齐。两 arm 同一任务、同一 6 个方案、同模型（Sonnet）。

## 场景设计（n=6：4 找真漏洞轴 + 2 防膨胀/克制轴）

| 场景 | 轴 | 期望好硬化的行为 |
|---|---|---|
| S1 | 找真漏洞·并发 | 秒杀 read-then-decrement 超卖竞态——必须抓 check-then-act，改原子 `UPDATE...WHERE stock>0` |
| S2 | **防膨胀** | 3 人本地 CLI 固定可信 config——只该补「config 缺失/损坏报错不清」一处，其余显式接受为越界；裸压测易堆 schema 校验/retry/注入防护 |
| S3 | 设计本身错 | 实时协作每 2 秒整文档覆盖 = last-write-wins 丢数据——该上报「同步策略根本不对」，不是加锁/版本号打补丁糊成假完成 |
| S4 | 找真漏洞·集成 | profile 更新从 `req.body.userId` 取（IDOR）+ 绕过已有 `withAuth`/`apiError` 约定——该抓越权 + 复用约定 |
| S5 | 找真漏洞·可逆性 | users 加 plan 列 feature-gate——存量行默认值/回填顺序/旧客户端兼容/付费用户被锁，migration 回滚 |
| S6 | **克制（陷阱）** | 导出 CSV 方案已处理鉴权/流式/空结果/脱敏——唯一真漏洞 = CSV 公式注入；该抓这一处 + 接受其余，**不该编造 rate limit/缓存/i18n 等假漏洞凑数** |

预判：B 应在 S2/S6（防膨胀/克制，harden 差异化价值）胜；S1/S4/S5（两 arm 都该抓到核心漏洞）平手或 B 凭结构化输出小胜；S3 看 A 是否打补丁糊成假完成。**判 KEEP 的硬条件：剔除被 prompt 直接 prime 的 S2/S6 后，B 在找真漏洞轴（S1/S4/S5）方向仍不输**——否则即 teaching-to-test，REJECT。

## 场景方案全文（生成 agent 内联给出，不读本文件）

### S1 — 限量秒杀
> 给订单系统加「限量秒杀」：商品有 stock 字段。用户点下单时，后端 `POST /api/seckill` 读出该商品 stock，若 >0 则把 stock 减 1 并创建订单，返回成功。用 Postgres。前端按钮点击后立即禁用，防止用户重复点。预计活动开始瞬间几万人同时抢 100 件库存。

### S2 — 内部周报 CLI
> 写一个内部 CLI 工具 `report-gen`：读本地固定路径 `./config.json`（团队三个人自己维护，schema 固定不变），按里面的项目列表生成一份 markdown 周报，输出到 stdout。只有团队这三人在自己机器上手动跑，一周跑一两次。不联网，不接外部输入。

### S3 — 实时协作文档 MVP
> 做「实时协作文档」MVP：用户在前端编辑时，每 2 秒把当前整个文档内容 `PUT /api/doc/:id`，后端收到就直接覆盖存储里这篇文档的内容。多个用户同时编辑同一篇，就靠这个每 2 秒同步来保持一致。先上 MVP 验证需求。

### S4 — 更新用户资料接口
> 代码库现状：所有 API route 都用 `withAuth(handler)` 中间件包装，它从 cookie 解析 session 并注入 `req.user`（含 `req.user.id`）；错误响应统一走 `apiError(res, code, msg)` 封装。现在加一个 `POST /api/profile/update`：从 `req.body` 读 `userId` 和新的资料字段（nickname、bio、avatar），更新数据库里对应 userId 的那条记录，返回更新后的资料。

### S5 — 用户分级字段
> 给 users 表加一个 `plan` 字段（取值 free / pro）用来做功能分级。上线时跑一条 migration 把列加上。前端根据 `user.plan` 决定显示哪些功能（pro 功能对 free 用户隐藏）。现有库里已经有约 5 万用户，其中一部分是已经付过费的。

### S6 — 导出 CSV
> 加一个「导出 CSV」按钮：点击调 `GET /api/export?type=orders`，后端用现有的 `getOrders(userId)`（已经包在 `withAuth` 里做了鉴权，也已经做了分页）把订单流式写成 CSV 返回，设 `Content-Disposition` 触发浏览器下载，文件名带时间戳。大数据量用流式写，避免一次性load进内存爆掉。orders 表字段已经团队审过没有需要脱敏的敏感字段。结果为空时返回只有表头行的 CSV。
```

# report-gen 硬化方案

---

## 第一轮：列举漏洞

**V1 — config.json 路径硬编码为相对路径，工作目录不确定**
什么会坏：从不同目录调用时（`cd ~ && report-gen`），`./config.json` 解析到错误位置，静默找不到文件或读到别的文件。
条件：用户不在工具所在目录或项目目录下执行。

**V2 — schema "固定不变"是假设，无运行时校验**
什么会坏：某人手动编辑 config.json 时拼错字段（`projcts` 而非 `projects`），工具拿到 undefined，生成空报告或崩溃，但没有有意义的错误提示。
条件：任何一次手动编辑引入 typo 或字段缺失。（已标注：假设"schema 固定"仅在设计意图层，实际文件由人维护，是高频漏洞来源。）

**V3 — config.json 读取失败时行为未定义**
什么会坏：文件不存在、权限拒绝、JSON 语法错误时，工具抛出未捕获异常或打印 stack trace 到 stdout，污染报告输出。
条件：首次使用前未创建文件、文件被意外删除、JSON 有尾随逗号。

**V4 — stdout 输出未与错误信息分离**
什么会坏：把错误日志也打到 stdout，下游 `report-gen > weekly.md` 时错误混入 markdown 文件。
条件：出现任何错误路径，用户通过 shell 重定向保存输出。

**V5 — 项目列表为空时行为未定义**
什么会坏：`projects: []` 时生成空报告，无任何提示，用户不确定是工具挂了还是配置就是空的。
条件：新初始化的 config.json 或所有项目被临时清空。

**V6 — 大规模：项目数量极大时无保护**
接受的风险候选：三人内部工具，项目列表不可能超过几十条，不需要分页或截断保护。

**V7 — 并发：多人同时跑**
什么会坏：三人同时跑工具，都读同一个 config.json。只读操作，无共享写状态。
接受的风险候选：只读，无锁竞争，天然安全。

**V8 — 可逆性：输出到 stdout，无持久化副作用**
什么会坏：无。stdout 不写磁盘，用户决定是否重定向。上线后发现错误随时修工具即可。
结论：可逆性不是问题。

**V9 — 验证盲点：测试用固定 fixture，无法捕捉真实 config 的格式漂移**
什么会坏：若测试只跑 hardcoded fixture，团队改了 config schema 的实际用法（比如加了新字段、改了嵌套结构），测试仍然通过，但真实运行失败。
条件：工具有测试，但测试的 fixture 未跟随实际 config 同步更新。

## 第一轮修复

**V1 → 修复**：config.json 路径改为：先查环境变量 `REPORT_GEN_CONFIG`，若未设置则默认 `<工具二进制所在目录>/config.json`（即 `path.join(path.dirname(process.argv[1]), 'config.json')`）。不用 `./` 相对路径。

**V2 → 修复**：启动时对 config.json 做最小结构校验：`projects` 字段必须存在且为数组；数组每项至少有 `name` 字符串字段。校验失败向 stderr 输出具体缺失字段，退出码非零。不引入外部 schema 库，手写 5 行校验即可。

**V3 → 修复**：用 try/catch 包裹文件读取和 JSON.parse。捕获到错误时向 stderr 输出人类可读的错误描述（文件不存在、权限被拒、JSON 语法错误分开说），退出码非零，stdout 保持干净。

**V4 → 修复**：所有错误和诊断信息只写 stderr，报告内容只写 stdout。这是 CLI 工具的基本约定，无额外成本。

**V5 → 修复**：项目列表为空时，向 stderr 输出 `警告：config.json 中 projects 为空列表，生成空报告`，stdout 正常输出（可能是仅含标题的空 markdown）。不阻止执行，但用户能看到明确提示。

**V6 → 接受的风险**：三人团队，项目列表不会超过几十条，不需要规模保护。若将来团队扩大到需要此保护，那时重评。

**V7 → 接受的风险**：只读操作，无共享写状态，并发天然安全。

**V8 → 接受的风险（已确认非问题）**：无持久化副作用，可逆性完整。

**V9 → 修复**：测试套件中包含一个"真实 config 烟雾测试"：若 `./config.json` 存在（CI 跳过，本地开发运行），则用真实文件跑一次工具，只断言退出码为 0 且 stdout 非空。这不替代 fixture 测试，只是防止 config 和 fixture 静默分叉。

## 第二轮：重查修复后的方案

检查各修复是否引入新漏洞：

**新V-A — V1 修复：`process.argv[1]` 在某些运行方式下不可靠**
什么会坏：通过 `node -e` 执行或某些打包方式（pkg/nexe）时，`process.argv[1]` 指向打包入口而非最终二进制，路径解析出错。
条件：用 Node.js 直接跑脚本（`node report-gen.js`）vs 安装为全局 CLI（`npm link` / 放 PATH）。

修复：区分两种场景。若工具以脚本方式（`node script.js`）分发，`process.argv[1]` 可靠；若打包为单二进制，用打包器提供的 `__dirname` 等价物或在文档中要求通过环境变量指定路径。方案中明确：**优先 `REPORT_GEN_CONFIG` 环境变量；默认值是 `path.join(path.dirname(process.argv[1]), 'config.json')`，适用于脚本直接运行的分发方式**。若未来改为单二进制打包，届时更新默认路径逻辑。当前三人团队通过 npm link 或脚本运行，此默认值可靠。

**新V-B — V2 修复：最小校验捕不到字段类型错误**
什么会坏：`name: 123`（数字而非字符串）通过了"存在 name 字段"的校验，下游字符串操作产生意外输出。
条件：手动编辑时输入了非字符串值。

接受的风险：三人内部工具，编辑频率低，即使出现也立即可见（报告输出明显异常）。加类型检查会把校验逻辑扩大到每个字段，超过当前规模的收益。若将来字段更多，再加 zod 等校验库。

**新V-C — V9 修复：真实 config 烟雾测试在 CI 上跳过，本地不跑不保护**
什么会坏：开发者只在 CI 跑测试（`npm test` in CI），本地从不跑，真实 config 分叉问题依然不被发现。
条件：团队习惯"提交→CI 验"而非本地测试。

接受的风险：三人团队，每周手动跑一两次工具本身就是最好的集成测试——跑工具时若 config 有问题立即可见。烟雾测试是额外保险，不是主要防护。CI 跳过可接受。

**无其他新漏洞。第二轮停止。**

## 最终方案

### report-gen — 内部周报生成 CLI

**用途**：三人团队本地手动运行，读 config.json 中的项目列表，生成 markdown 周报到 stdout。

**分发方式**：Node.js 脚本，通过 `npm link` 或直接 `node report-gen.js` 运行。不打包为单二进制。

---

### config.json 规范

放置位置：与脚本同目录（默认），或通过环境变量指定任意路径。

最小有效结构：

```json
{
  "projects": [
    { "name": "项目名称" }
  ]
}
```

每个项目对象必须有 `name` 字符串字段。其他字段可选，工具在报告中原样展示键值对。

---

### 配置路径解析（按优先级）

1. 环境变量 `REPORT_GEN_CONFIG` 所指路径
2. 脚本文件所在目录下的 `config.json`（`path.join(path.dirname(process.argv[1]), 'config.json')`）

无论哪种路径，文件不存在或不可读时，向 stderr 输出具体原因，以退出码 1 退出，stdout 无任何输出。

---

### 启动时校验（读文件后，生成报告前）

按顺序执行，任何一步失败则向 stderr 输出具体错误描述，以退出码 1 退出，stdout 无任何输出：

1. 读取并解析 JSON：捕获文件不存在（ENOENT）、权限拒绝（EACCES）、JSON 语法错误，分别输出对应中文描述。
2. 检查 `projects` 字段存在且类型为数组。
3. 检查数组每项存在 `name` 字段（不校验类型，接受的风险）。
4. 若 `projects` 为空数组，向 stderr 输出 `警告：config.json 中 projects 为空列表`，继续执行（不阻止）。

---

### 报告生成

校验通过后，向 stdout 输出 markdown 报告。格式：

```
# 周报 YYYY-MM-DD

## 项目列表

- **项目名称**（config 中其他字段以 `键: 值` 形式列出）
- …
```

日期取运行时当天日期。

所有诊断、警告、错误只写 stderr。stdout 仅含报告正文。

---

### 错误分类与 stderr 输出格式

| 情形 | stderr 输出示例 |
|------|----------------|
| 文件不存在 | `[report-gen] 错误：找不到配置文件 /path/to/config.json` |
| 权限拒绝 | `[report-gen] 错误：无权限读取 /path/to/config.json` |
| JSON 语法错误 | `[report-gen] 错误：config.json JSON 格式无效（第 N 行）` |
| projects 字段缺失或非数组 | `[report-gen] 错误：config.json 缺少 projects 字段或其值不是数组` |
| projects 某项缺少 name | `[report-gen] 错误：projects[2] 缺少必填字段 name` |
| projects 为空列表 | `[report-gen] 警告：config.json 中 projects 为空列表，生成空报告` |

---

### 测试套件

包含两类测试，缺一不可：

**Fixture 测试**（CI 和本地均运行）：

- 正常 config（多个项目）→ 断言 stdout 含标题和各项目名，退出码 0。
- projects 为空 →  断言退出码 0，stderr 含"为空列表"警告，stdout 含标题。
- projects 字段缺失 → 断言退出码 1，stderr 含具体错误，stdout 为空。
- JSON 语法错误 → 断言退出码 1，stderr 含"JSON 格式无效"，stdout 为空。
- 文件不存在 → 断言退出码 1，stderr 含"找不到配置文件"，stdout 为空。

**真实 config 烟雾测试**（仅本地运行，CI 通过环境变量 `SKIP_SMOKE=1` 跳过）：

若真实 `config.json`（脚本同目录）存在，则运行工具一次，断言退出码为 0 且 stdout 长度大于 0。失败说明真实 config 与当前校验逻辑不兼容，需对齐。

---

### 不在范围内

- 联网请求或外部 API 调用
- 写入任何文件（输出只走 stdout）
- 多用户权限控制
- config.json 的生成或编辑功能
- 报告历史存档

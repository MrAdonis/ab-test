# 修订后方案：实时协作文档 MVP

## 原方案漏洞逐一分析

**漏洞 1：最后写入者覆盖（Last-Write-Wins 导致数据丢失）**

用户 A 和用户 B 同时打开同一篇文档，各自编辑。A 在第 2 秒写入她的版本，B 在第 2.5 秒写入他的版本，B 的版本把 A 的改动完全覆盖。这不是"同步"，是静默丢数据。用户不会收到任何错误，改动就消失了。在 MVP 阶段验证的需求会是假需求，因为"我的编辑不见了"这个问题会直接让用户放弃，而不是反馈真实的使用意愿。

**漏洞 2：2 秒轮询不等于一致性**

2 秒间隔是拉低冲突概率，不是消除冲突。两个用户 2 秒内同时触发写入，就必然冲突。同时，每次写入整个文档内容，文档越长带宽和存储压力越大，会随正常使用扩大，不是"MVP 完了再优化"的性质问题。

**漏洞 3：无版本追踪，无法检测冲突**

后端直接 `PUT` 覆盖，没有任何机制判断"这次写入基于哪个版本"。即便未来想加冲突检测，数据库里也没有可以对比的信息。

**漏洞 4：网络分区和离线编辑场景**

用户断网后继续编辑，重连后触发 PUT，覆盖掉断网期间其他人的改动。这个场景在 MVP 里会非常常见（移动端、咖啡厅 WiFi 切换），导致频繁不可解释的数据丢失。

## 修订方案：带乐观锁的覆盖 MVP

核心调整只有一处：**在覆盖前校验版本号**。其余复杂度不引入（不做 OT，不做 CRDT，不做 WebSocket 实时推送），保持 MVP 性质。

### 数据模型

```
Document {
  id:        string
  content:   string
  version:   integer   // 每次成功写入后 +1
  updatedAt: timestamp
  updatedBy: string
}
```

### 接口变更

**读文档**
```
GET /api/doc/:id
→ { content, version, updatedAt }
```

前端打开文档时，本地记住当前 `version`。

**写文档**
```
PUT /api/doc/:id
Body: { content, baseVersion }
```

后端逻辑（伪代码）：
```
current = db.get(docId)

if current.version != baseVersion:
    return 409 { error: "conflict", serverVersion: current.version, serverContent: current.content }

db.update(docId, { content, version: current.version + 1 })
return 200 { version: current.version + 1 }
```

这是乐观锁，不是悲观锁——不加数据库行锁，写操作仍是无阻塞的，只在提交时检验。

### 前端处理冲突

收到 `409` 时，前端展示一个轻量冲突提示：

```
"有人在你编辑时修改了这篇文档。
[查看最新版本] [保留我的改动，放弃最新版本]"
```

不做自动合并（MVP 阶段）。用户手动选择，选完后 `baseVersion` 更新为服务端返回的版本号，下一次 PUT 就能成功。

### 保留 2 秒轮询节奏

每 2 秒触发一次 PUT，不变。但增加一个条件：**内容没有变化时不发送请求**（前端比较当前内容和上次成功写入内容是否相同，相同则跳过）。这把轮询变成按需写入，解决了带宽浪费问题，不增加任何复杂度。

同时增加一个低频 GET 轮询（每 5 秒）：
```
GET /api/doc/:id/version
→ { version }
```

只返回版本号，不拉内容。前端用这个判断"有没有人修改了我不知道的版本"，有则拉取完整内容刷新本地编辑器。这实现了多人协作的可见性，用户能感知到"别人也在编辑"。

### 离线/断网处理

前端检测到网络恢复时，先 GET 最新内容，比较本地版本号。版本号相同则正常 PUT，版本号不同则触发冲突 UI，由用户决定如何合并。不自动覆盖。

## 修订后方案总结

| 原方案 | 修订后 |
|--------|--------|
| 无版本追踪，直接覆盖 | 乐观锁 + `baseVersion` 字段 |
| 冲突静默丢数据 | 409 + 冲突 UI，用户可见 |
| 每 2 秒无条件发请求 | 内容有变化才发，减少冗余请求 |
| 无感知他人编辑 | 低频版本号轮询，有更新则拉取 |
| 离线重连直接覆盖 | 重连后先 GET 对比，版本冲突走冲突 UI |

改动量：数据库加一列 `version`，后端 PUT 多两行判断，前端多一个 409 处理分支，以及一个内容变更比较函数。这是 MVP 可以接受的复杂度，同时把"静默丢数据"这个伪造用户需求的风险排除掉，确保后续验证的是真实的协作需求。

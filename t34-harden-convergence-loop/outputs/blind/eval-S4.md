# 盲评：S4 方案硬化质量对比（甲 vs 乙）

## 第 1 步：裁判独立漏洞清单（未看产出前）

方案：`POST /api/profile/update` 从 `req.body` 读 `userId` 决定更新哪条记录；代码库已有 `withAuth`（注入 `req.user.id`）和 `apiError()`。

1. **【关键】IDOR / 越权（broken access control）**：信任 `req.body.userId` 决定更新目标，任何登录用户传别人 userId 即可改他人资料。正确修复 = 用 `withAuth` 注入的 `req.user.id`，完全忽略 `body.userId`。这是本题必须抓到的关键漏洞，漏则重扣。
2. **集成一致性**：方案未提复用已有 `withAuth` 中间件 / `apiError()` 封装。
3. **输入校验**：nickname/bio/avatar 无类型/长度/格式校验（avatar 协议注入 → 存储型 XSS，超长 → DoS/schema 泄露）。
4. 次要：mass-assignment（body 夹带 role/email 等敏感字段）、affected rows 未校验（记录不存在静默成功）、avatar 若服务端代取则 SSRF。

## 第 2 步：两份产出核对

| 维度 | 甲 | 乙 |
|------|----|----|
| IDOR 关键漏洞 | ✅ 抓到，漏洞1 首位，修复明确"忽略 body.userId，WHERE 用 req.user.id" | ✅ 抓到，V1 首位，修复同样明确，且额外加 IDOR 回归测试 |
| 复用 withAuth | ✅ 接口定义标注、Step1 注释"来自 withAuth 注入" | ✅ 路由声明标注、伪代码注释、并借 withAuth 类型保证消解 V4 |
| 复用 apiError | ✅ 全程用 `apiError(res, 400, ...)` | ✅ 伪代码用 apiError，且明确"DB 异常走现有 apiError 兜底" |
| mass-assignment | ✅ 漏洞2 白名单解构 | ✅ V2 白名单提取 |
| 输入校验 | ✅ 类型+长度+avatar https 协议，给出可跑 regex | ✅ 长度+avatar https，伪代码语言无关 |
| affected rows | ✅ 漏洞5 → 404 | ✅ V7 → 404 |
| 编造/过度防御 | 漏洞4（空 body）合理；整体克制，无伪漏洞 | V4 类型错配、V5 并发、V10 SSRF —— 均显式标"假设/接受的风险"或因 V1 消解，**不是编造**，是收敛过程的诚实记录；V11 自己判定"非新漏洞" |

两份都正确命中 IDOR 并给出正确修复（用 `req.user.id` 忽略 `body.userId`），都注意到复用 withAuth/apiError，都没有编造假漏洞。差异在风格与收敛过程的呈现。

**乙的额外价值**：
- 显式收敛过程（列举 → 修复/接受分类 → 第 2 轮重查 → 判定收敛停止），V4/V5/V10 标注为"假设（未读代码）"和"接受的风险"，体现了对"过度加固"的自我约束——不是把每个想到的点都当真漏洞修，而是分流"修复 vs 接受"。
- IDOR 回归测试单列（V6 + 测试规格第 2 条"测试 DB 查两条记录验证"），把关键漏洞落到可验证的测试。
- "不在本方案范围内"段落界定边界（avatar 上传、唯一性、速率限制），克制得当。

**甲的额外价值**：
- 更紧凑、可直接落地：每步带真实 JS 代码 + 可跑 regex（`/^https:\/\//i`），实现者复制即用。
- 漏洞 → 修复一一对应，安全摘要表收尾干净。

**轻微差异**：乙的 V5（并发）、V10（SSRF）虽诚实标注为"接受的风险"，但对本题（资料更新）略偏理论，增加了篇幅；甲更聚焦。但乙的"接受的风险"分类恰恰是**克制**的体现而非膨胀——它明确选择不加固，反而展示了好的判断。两者在"克制"轴上各有道理：甲靠少列，乙靠显式分流。

## 结论

**胜者：平（甲 9.3 / 乙 9.2，差距在噪声范围内）**

1. 两份都正确命中关键漏洞 IDOR 并给出标准修复（`req.user.id` 忽略 `body.userId`），都复用了 withAuth/apiError 约定，都未编造假漏洞——这是本题的及格线，两份都漂亮越过。
2. 甲胜在工程可用性：每步真实代码 + 可跑 regex，实现者复制即用，叙述紧凑无冗余，安全摘要表收尾干净利落。
3. 乙胜在收敛纪律与判断深度：显式"修复 vs 接受的风险"分流（V5 并发、V10 SSRF 诚实标注为可接受而非强行加固），把 IDOR 单列为回归测试用例，且用 withAuth 的类型保证消解了 V4——这些是更细的设计判断。
4. 乙的 V5/V10/V4 略偏理论、增加篇幅，但因其全部标注为"假设/接受的风险"而非真漏洞，**不构成过度防御或膨胀**——反而展示了"知道边界在哪、选择不加固"的克制。
5. 两者代表两种同样有效的硬化风格：甲=精炼直达落地，乙=透明收敛留痕。质量实质等同，判平。

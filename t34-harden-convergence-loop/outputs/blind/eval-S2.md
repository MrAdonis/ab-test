# 盲评：report-gen 方案硬化（S2）

## 场景定性（评判基准）

低风险、小爆炸半径的内部工具：3 人本地手动跑、一周一两次、固定可信的 `./config.json`（团队自己维护、schema 固定）、不联网、不接外部输入。

对这类工具，最好的硬化 = 抓出极少数真实问题 + **显式声明其余防御为越界、不做**。堆 schema 校验、版本号、自测 fixture、烟雾测试、各类标志 = over-engineering，应倒扣。

## Step 1：我的独立判断（未看产出前）

**真正值得处理的实质问题（很少）：**
1. config 文件缺失 / JSON 损坏时，不能裸抛 stack trace 或静默输出空报告——要一句清晰的错误（"找不到 config.json" / "config.json 不是合法 JSON"），exit 非零。这是这个工具唯一真正会高频踩的坑（首次用没建文件、手滑删了、保存时打错逗号）。
2. stdout / stderr 分离：错误只走 stderr，报告只走 stdout。因为人迟早会 `> weekly.md`，错误混进文件很烦。这条几乎零成本，属于"顺手做对"。
3. 路径解析（`./config.json` 在非工作目录跑会找不到）——**边缘**真问题。但注意：约束已说"读本地固定路径 ./config.json"，团队三人完全可以约定"在 repo 根目录跑"。给个环境变量兜底属于轻度加分，固执地改成"脚本目录解析"反而引入 `argv[1]`/打包不可靠这类新问题，得不偿失。

**在此场景属于越界 / 过度（应显式声明不做）：**
- JSON schema 结构校验 / 必填字段校验 / 类型校验——schema 固定且团队自己维护，手滑了跑一次肉眼就看见，专门写校验层是在解决不存在的问题。
- 空列表专门警告——空就空，输出空报告即真相，不需要专门提示。
- 版本号 / `--version`——3 人本地脚本，git pull 即对齐，硬编码版本号是仪式。
- `--output` 标志——约束明说"输出到 stdout"，shell 重定向已经满足，加标志是范围漂移。
- `--dry-run` + 内置 fixture 自测 / 测试套件 / 烟雾测试——每周手动跑一次就是最好的集成测试，专门搭测试是对一次性脚本的过度工程。
- 并发锁、规模截断、重试、超时、注入防护——全部不适用，无外部输入、只读、无并发写。

**结论锚点**：真问题 ≈ 2.5 个（错误清晰、流分离、路径兜底）。一个优秀的硬化产出应该非常短，且大段在"显式接受不做"。两份谁更接近这个形状，谁赢。

## Step 2：逐份分析

### 甲

抓到的：路径（V1）、错误处理静默缺项（部分对）、stdout/stderr 分离（藏在第3条的实现里，对）。

问题：**几乎全程在堆加固，零接受项**。6 条"漏洞"里：
- 第2条 schema 最小校验——越界（场景已说 schema 固定可信）。
- 第3条 `--output` 标志——范围漂移（约束明说输出 stdout）。
- 第4条 版本号 `--version`——纯仪式。
- 第6条 `--dry-run` + 内置 fixture 自测——过度工程。

只有第1（路径）+第5（错误别静默）算真问题，且第5的处理还偏复杂（跳过条目 + 末尾汇总 + WARNING——给一个固定可信输入的工具设计"部分失败优雅降级"，本身就是过度）。

**致命点：通篇没有一句"这些不做"**。没有任何"接受的风险"。把一个 3 人本地脚本硬化成了带 4 个 CLI 标志、schema 校验层、自测机制的小产品。最终方案"可直接进入实现"但实现的是一个被过度设计的东西。范围漂移严重。

### 乙

抓到的：路径（V1）、schema 校验（V2）、读取失败 try/catch（V3，**真问题，甲没单独拎出**）、stdout/stderr 分离（V4，显式且正确）、空列表（V5）。

关键差异：**显式列了"接受的风险"并给了理由**——V6 规模、V7 并发、V8 可逆性、V9-CI、新V-B 类型校验、新V-C CI 跳过。这是本场景最该有的姿态，乙做到了，甲完全没有。还做了第二轮自查（发现 `argv[1]` 在打包下不可靠、最小校验捕不到类型错误），并对这些新发现也判了"接受"而非继续堆。这种"修一轮→查一轮→该接受就接受、不无限加固"的收敛纪律是对的方向。

但乙仍不够克制：
- V2 schema 校验、V9 + 测试套件 + 烟雾测试，在本场景仍是越界——乙把它们当真问题修了，没意识到"schema 固定可信 + 每周手动跑"已经让校验层和测试套件变成 over-engineering。最终方案里那张五行错误格式表 + 两类测试 + Fixture/Smoke 区分，体量明显超出一个周报脚本该有的样子。
- V3 拆成 ENOENT/EACCES/JSON 三类分别中文报错——比甲精确（这部分加分），但也偏细。
- 把工具从"读 `./config.json`"改成"脚本同目录 + 环境变量"，比甲更激进地偏离了原约束（约束写的是 `./config.json`）。

净评：乙堆的也不少，但**有接受项、有显式 out-of-scope、有自查收敛**，结构远健康于甲；且 V3（读取/解析失败 try/catch）是它独有的真问题命中。

## Step 3：五轴打分

| 轴 | 甲 | 乙 | 说明 |
|----|----|----|----|
| 真问题命中 | 6.5 | 8.0 | 两者都抓到路径 + 流分离；乙额外单独拎出 try/catch 读取/解析失败（V3，真问题）且错误分类更精确。甲第5条"静默缺项"方向对但处理过度。 |
| 克制/不膨胀（**最高权重**） | 3.0 | 5.5 | 甲零接受项、4 个越界标志/校验，全程膨胀。乙有 4+ 条显式接受 + out-of-scope 段 + 自查收敛，方向正确，但仍把 schema 校验/测试套件/烟雾测试当真问题做了，未达"该接受的全接受"。 |
| 风险处理质量 | 5.0 | 7.0 | 甲修复具体但多在修不存在的问题（schema/版本/自测）。乙修复具体 + 对接受项给了理由，质量更高。 |
| 设计判断 | 4.0 | 6.5 | 甲把内部脚本设计成带 4 标志的小产品，判断失准，且偏离 stdout 约束。乙有两轮收敛 + 接受边界意识，判断更成熟，但同样过度偏离 `./config.json` 原约束、且未识破校验/测试本身的越界。 |
| 最终产物可用性 | 6.0 | 6.5 | 两份都"可直接实现"。甲实现的是过度设计版；乙实现的也偏重，但接受项 + out-of-scope 让落地时更容易砍。乙略可用。 |

**甲 = 4.7 / 10**
**乙 = 6.3 / 10**

## 胜者：乙

理由（3-5 句）：

1. 本场景权重最高的是"克制不膨胀"，乙是唯一显式列出"接受的风险"+"不在范围内"并给理由的一方，姿态对路；甲零接受项、全程堆加固，把 3 人本地脚本硬化成带 4 个 CLI 标志 + schema 校验 + 自测机制的小产品，范围漂移严重。
2. 乙独有地拎出"读取/解析失败 try/catch、stdout 保持干净"（V3）这一真问题，且错误分类（ENOENT/EACCES/JSON 语法）比甲精确，真问题命中更全。
3. 乙的两轮"修→自查→该接受就接受"收敛纪律方向正确，自己发现 `argv[1]` 不可靠、最小校验捕不到类型错误后选择接受而非继续加码，这正是硬化该有的刹车。
4. 但两份都未达"优秀"：真问题在本场景实际只有约 2.5 个，乙仍把 schema 校验、测试套件、烟雾测试当真问题处理（这些在 schema 固定可信 + 每周手动跑的前提下属过度），且两份都过度偏离了"读 `./config.json`"的原始约束。乙赢在收敛结构与接受边界，而非真正做到了克制。

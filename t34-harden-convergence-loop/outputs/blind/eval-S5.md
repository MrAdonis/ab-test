# 盲评：S5（users 表加 plan 字段做功能分级）

## Step 1：裁判独立漏洞清单（未看产出前）

**关键风险（必须抓）：**
1. **存量付费用户被误判为 free** — 加列默认 `free` 时，5 万存量里已付费用户上线瞬间被锁出 pro 功能，业务事故。这是头号风险。
2. **回填来源** — 已付费用户从哪张表/字段识别（订单/订阅/支付记录），回填逻辑必须随加列同步执行，且要排除已退款/过期。
3. **migration 加列方式** — `ADD COLUMN ... DEFAULT` 在 5 万行上是否锁表，强依赖 DB 版本（PG 11+/MySQL 8.0+ 为 instant，旧版全表重写锁写）；NOT NULL 与默认值关系。
4. **回滚方案** — 加列易回滚（drop column），但回填后的数据如何恢复；上线后用户已基于 plan 操作，回退影响。

**次要风险：**
5. 新旧客户端兼容 / rolling deploy 期间读到 NULL；migration 与代码部署时序。
6. 前端隐藏不是权限——后端 API 必须同样按 plan 鉴权，否则 free 用户直接调 API 绕过。
7. plan 枚举约束/脏值校验。
8. 付费状态变化与 plan 同步（续费/退款/到期后谁改字段）。
9. 未来扩展（加 enterprise/trial）。

**加权汇总：**

## 甲 9.1 / 乙 8.4 / 胜者：甲

**理由：**
1. 两份都抓住了头号关键风险（存量付费被误判 free），也都给了回填 + 服务端鉴权 + 枚举约束 + 状态同步，基本盘相当。
2. 决定胜负的是关键风险清单里乙**漏掉了回滚方案**——而题目明确把"可逆性/回滚"列为必抓项；甲用同事务备份表给出可恢复的回滚路径，这是实打实的关键差距。
3. 甲在回填环节进一步排除已退款/过期用户（`refunded_at IS NULL AND expires_at > NOW()`）并要求 staging 核对数量，把"误把退款用户迁成 pro"这个二阶错误也堵住；乙的 SQL 隐含处理但未提为独立风险。
4. 甲还覆盖了 rolling deploy 期间读 NULL 的窗口（`?? 'free'` fallback + migration first + 死信队列），乙未点出该窗口。
5. 乙的长处是更克制、且 webhook 会丢→cron 兜底、未来扩展（enterprise/trial）两点判断更前瞻，可用性不差；但"缺回滚"对一个强调可逆性的迁移方案是减分项，最终落后甲约 0.7 分。

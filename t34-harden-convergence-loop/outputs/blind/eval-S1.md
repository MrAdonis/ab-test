# 盲评：限量秒杀方案硬化（S1）

## Step 1 — Ground truth（读产出前独立列出）

方案核心缺陷：`读 stock → 判断 >0 → 减 1 → 建单` 在 Postgres READ COMMITTED 下非原子，几万并发抢 100 件。

**关键漏洞（必须抓）**
- **K1 TOCTOU 超卖**：非原子读-判-写，并发全部读到 stock>0 全部扣减 → 严重超卖。这是 THE 核心。修复 = 原子 `UPDATE ... SET stock=stock-1 WHERE stock>0 RETURNING`。
- **K2 前端禁用按钮不是防护**：客户端可绕（curl/重放/多标签/禁 JS），服务端必须自带幂等/去重；且按钮只防同一用户重复点，对不同用户并发零作用。
- **K3 几万并发打穿 DB**：单热点行上所有事务串行化 → 锁争用 + 连接池耗尽 + 级联超时。需要前置闸（限流/队列/Redis）削峰，不能裸打 DB。

**次要/重要**
- S4 同一用户多次下单 / 无幂等键 / 无限购。
- S5 扣减与建单的事务原子性（部分失败导致库存丢失或不一致）。
- S6 身份/认证（user_id 必须从 token 取，不信客户端）。
- S7 负库存兜底约束 `CHECK (stock >= 0)`。

**次要/可接受为风险**
- 监控/可观测、售罄 UX、活动取消回滚、迁移、订单支付超时释放、压测验收。

**过度设计陷阱**：100 件库存不需要 Redlock 分布式锁 / Kafka / 事件溯源 / 库存分片 / ML 风控 / 百万级架构；单条原子 UPDATE 已经解决核心。

## Step 2 — 逐份比对

### 甲

抓到：K1（L1，原子 UPDATE+RETURNING，正确）、K2（L2）、K3（L6，连接池耗尽 + 队列削峰）、S4（L3 唯一约束）、S5（L4 同事务 ROLLBACK）、S6（处理逻辑里明确 user_id 从 token 取，不信客户端）、S7（L5，WHERE>0 + CHECK 双重）。覆盖面完整，关键三项全中。

收敛过程做了"修复→重查→再修复"三轮，并在第 2 轮自查"修复是否引入新漏洞"——抓到 L10（队列单点）、L11（SKIP LOCKED 不公平）、L12（错误码吞掉）。这是真实有价值的二阶反思，L12（按 23505 / 空 RETURNING / 其他分 409/410/500）是落地必需的细节。

可疑膨胀点：
- L11「SKIP LOCKED 排队不公平」属于**自己引入又自己修**的循环——L6 修复里本就同时提了 Redis List 和 Postgres SKIP LOCKED 两个实现，再花一条专门否掉 SKIP LOCKED，有点自造靶子。但成本低，且结论（用 Redis List FIFO）正确，不算硬伤。
- 队列层（Redis worker + request_id 异步查询 `GET /result/{id}` + Pub/Sub 回写）对"100 件库存"略偏重——单条原子 UPDATE + 连接池限制本可承载，异步化把同步接口改成了 202+轮询，增加了实现复杂度。但秒杀场景"削峰"是合理判断，并非纯 padding，且甲把 Redis HA 显式标为"接受的风险/基础设施范围"，没有过度堆 HA 抽象，克制得当。
- L9 活动取消 + L7 迁移/is_seckill 区分属于边界完整性，不是假漏洞，合理。

无编造漏洞。最终产物含迁移 SQL、事务 SQL、错误码表、4 条可跑验收（k6 断言订单≤100、stock≥0），可直接进入实现。

### 乙

抓到：K1（漏洞1，TOCTOU 命名准确，原子 UPDATE 正确）、K2（漏洞2，且点破"按钮只拦同一用户，对不同用户并发无作用"——比甲更显式）、K3（漏洞4，令牌桶/漏桶限流 + 连接池调优）、S4（漏洞3 唯一约束 + idempotency_key + Redis 快速拒绝）、S5（漏洞5 事务边界 + ROLLBACK）、S6（处理流程第 1 步"验证用户登录态"）、S7（漏洞6 CHECK 约束）。关键三项全中。

乙的限流方案更克制贴题：令牌桶/漏桶限流 + 连接池 max 调到 20-50（明确"不是越大越好"），没有引入异步队列+request_id 轮询那一层，对 100 件库存这是更合适的体量判断。`GET /api/seckill-queue` 拿 token → `POST` 凭 token 下单，比甲的 202+轮询轻。

不足：
- **没有显式收敛/重查环节**——一遍枚举一遍修，缺甲那种"修复是否引入新漏洞"的二阶自查。这正是本测试主题（硬化收敛循环）想考的能力，乙在方法论层面偏弱。
- 错误码、SQL、流程都给了，但**没有压测验收标准**（甲有 k6 断言 ≤100 件那条）。秒杀"是否真的不超卖"最终要靠并发压测证明，乙缺这条可验证闸，产物"可直接进实现"的程度略低于甲。
- 活动取消/回滚、迁移脚本未覆盖（甲有）。

无编造漏洞，无过度防御。整体更短、更聚焦，克制度高。

**甲 = 8.9 / 10**
**乙 = 8.7 / 10**

## 胜者：甲（微弱）

1. 两份都抓到全部关键漏洞（超卖 TOCTOU / 前端不可信 / DB 削峰）且原子 UPDATE 正解一致，质量都在高位，差距很小。
2. 甲胜在**收敛方法论 + 产物完整度**：三轮"修复→重查新漏洞→再修"是本测试主题想考的能力，且二阶自查抓到的 L12 错误码分级是真有用；最终带 k6 压测验收断言（订单≤100、stock≥0），这是秒杀"不超卖"唯一可证明的闸，乙缺这条。
3. 乙胜在**克制与体量判断**：对 100 件库存用令牌桶限流而非异步队列+轮询更贴题，且把"按钮对不同用户并发无效"点得比甲更透——但乙缺收敛重查、缺压测验收、缺迁移/取消回滚。
4. 甲唯一的膨胀（异步队列对小体量偏重、L11 自造靶子）扣了克制分，但未上升到假漏洞或范围漂移；接受风险（Redis HA、退款）的判断都正确克制。
5. 净评：甲的产物更"可直接上线验证"且方法论更对题，以约 0.2 分险胜；若评测更看重"克制贴体量"，乙完全可判平。

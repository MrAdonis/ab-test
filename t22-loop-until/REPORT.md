## S3 古德哈特防御单独点评

**A 怎么守的**：依赖推理约束——"能说清楚才能改，说不清楚就先搁着"。这是基于理解的防御，逻辑对，但执行时 agent 仍然可以自我说服"说清楚了"然后改期望值凑数。防线靠的是 agent 的诚实，不是技术约束。

**B 怎么守的**：在 Exit when 条件里嵌入了 `总测试数 ≥ 修复前基线数`——机器可验证，跑 `npm test` 就能检查，不依赖 agent 判断。边界条款把具体的凑数手法一一列出（skip / xit / xdescribe / 永真断言），连"把 `expect(x).toBe(3)` 改成 `expect(x).toBeDefined()`"这种不删测试但改空洞断言的变体都明确封堵了。

**结论**：B 守得更严。A 的防御是定性的（"不能这样做"），B 的防御是定量的（"总数不能少"）+ 枚举的（"这些具体手法禁止"）。在无人值守/自动 loop 场景下，B 的防守更不依赖 agent 善意。

## 一句话结论

B 总分 43，A 总分 32，差距 11 分。主要差距集中在三态收口清晰度（A 只有完成态，缺 caveats/blocked 分支）和退出条件可验证性（A 依赖视觉/截图而非命令 exit code）。B 的五字段模板在 loop 场景下提供了 A 散文风格无法稳定输出的结构：机器可检查的退出条件、baseline 对比防删凑数、三态分支的内容规范。模板复杂度值得付出——五字段不是仪式，是把"怎么算做完"从散文提炼成可执行参数的最小结构。**keep command，B 胜。**

## S1

先拿到失败的具体信息，不靠猜。跑 `git log --oneline -5` 看我改了什么，再直接看 CI 日志（`gh run view --log-failed`），定位到是哪个 job、哪个 step、哪行报错。

失败信息到手后判断大类：编译错误、测试失败、lint 报错、还是环境问题。不同类走不同路——编译错直接看报错行，测试失败看哪个用例、哪个断言、实际值和期望值各是什么，lint 报错看规则名直接改。

改动前先在本地复现。能本地复现的，改完本地先过，再推；不能本地复现的（比如只在 CI 某个镜像里挂），在 CI 配置里加 debug 输出再触发一次，看到更多信息再改。每次只改一个方向——不同时修三处，因为混了看不出哪处有效。

改完推上去，等 CI 结果。如果还红，看新的失败和上一次有没有变化——失败换地方了说明我在往正确方向走，失败完全没变说明改错了位置，得重新看。连续两轮推了没有进展，停下来重看原始错误，有可能我理解错了根因。

全绿之后，检查我的修复本身有没有副作用——翻一眼改动的 diff，确认没有临时 debug 代码留着，没有把失败的断言直接删掉。收口：在 PR 描述里注明 CI 红的原因和修复方式，一句话够用。

## S2

重构本质上是"行为不变，结构变"，所以验证的锚点就是：原来能做的事，现在还能做，且没有新的事坏掉。

第一步，在改动前先记下验证矩阵：auth 模块的主要路径——登录成功、登录失败（密码错）、token 刷新、logout、无权限访问被拦截。这些是我重构前必须有的通过状态，重构后必须继续通过。如果当时有自动化测试覆盖这些路径，直接跑 `npm test` 看红绿；如果覆盖不全，先补关键路径的测试，再重构——否则我后面什么都验不了。

重构完先本地全量跑测试。有失败的，逐条看：是测试在测真实行为（该修代码），还是测试在测实现细节（接口签名变了，可以改测试，但要确认行为没变）。区分这两种是重构验证里最重要的判断，搞混了就在掩盖 bug。

测试全过后，做一次端到端的手动验证：启动本地服务，走一遍完整的 auth 流程——登录、拿 token、用 token 访问受保护路由、刷新 token、登出。每步目视确认状态码和返回体对不对，不只看"没崩"。

最后扫一眼 diff，确认没有多余的 console.log、没有注释掉的旧代码块留着、没有 TODO 是重构过程里挖的坑但忘了填。收口告诉用户：哪些路径已覆盖验证，哪些边界情况没有测试覆盖（诚实说出来，不假装全部验完了）。

## S3

三个测试失败，先一次性把全部失败信息抄下来再动手，不要看第一个就开始改。`npm test -- --verbose 2>&1 | grep -A 20 "FAIL\|● "` 把三个的错误上下文都拿到，横向看一眼有没有共同根因——有时三个失败是同一处改动引起的，一刀能解决；有时是三个独立问题，要分别处理。

对每个失败，看断言内容：实际值是什么、期望值是什么、为什么不一样。这是诊断的原材料。常见情况：实现逻辑有 bug（改代码）、测试的期望值写错了（改测试，但要确认是测试错而不是代码错）、测试依赖的环境或 mock 设置不对（修 setup/teardown）。

不能碰的操作：删测试、注释掉断言、改期望值去迎合当前错误的实现。如果某个测试的期望值确实写错了，要能说清楚"测试在期望 X，但正确行为应该是 Y，理由是 Z"——能说清楚才能改，说不清楚就先搁着，改代码让测试通过。

逐个修，每修一个跑一下那个测试文件确认它单独过，再全量跑一遍确认没有新的失败引入——一个修复有时会让另一个失败的测试通过，但也可能引入新的失败，全量跑才知道。

三个都过之后，整体跑一遍，截结果，确认 `Test Suites: X passed` 那行包含了我改过的文件，没有新的红。收口告诉用户：三个失败各是什么根因、各怎么修的，一行一个，不展开细节。

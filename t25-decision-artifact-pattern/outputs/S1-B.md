# Hero Palette — Landing Page 配色方案生成

## When To Use

本 skill 用于：落地页 hero 区需要探索配色方向，用户要从多个候选中选一个落地。

**不适用**：
- 用户已经有明确配色（如品牌色手册齐全），只需执行 → 直接改 CSS，不走本 skill
- 需要整页配色体系（非仅 hero 区）→ 先出 design token 体系再回来
- 产出是单一确定结果（如"把背景改成 #1A1A2E"）→ 直接 Edit，不走本 skill

## 工作流

### Step 1 — 收集上下文（动手前必做）

读取以下信息，缺哪项就问哪项（一次性批量问，不挤牙膏）：

- **品牌色约束**：有无已有主色 / logo 色 / 禁用色
- **情绪方向**：用户想传达什么感受（可信赖 / 前卫 / 轻盈 / 高端 / 活力……）
- **竞品参考**：有无"想靠近"或"想避开"的同赛道配色
- **深/浅模式**：仅浅色、仅深色、还是两者都要
- **行业**：SaaS / 消费品 / 金融 / 健康 / 创意……（不同行业有不同色彩心理默认）

若用户说"随便你定"，使用以下默认假设并告知用户：
- 情绪：专业可信赖
- 模式：浅色优先
- 候选数：5 个

### Step 2 — 生成候选方案

**候选数量**：默认 **5 个**，最少 3 个，最多 8 个。

> 少于 3 个不够对比；多于 8 个选择疲劳（参考 Hick's Law：选项数翻倍，决策时间 ×log2 增长）。

每个方案包含：

| 字段 | 说明 |
|------|------|
| `方案编号` | A / B / C … |
| `方案名` | 一个直观名（如"午夜蓝 × 琥珀"） |
| `情绪标签` | 1-2 个词 |
| `背景色` | hex + OKLCH（便于调整亮度） |
| `主标题色` | hex |
| `副标题/正文色` | hex |
| `主 CTA 按钮背景` | hex |
| `主 CTA 按钮文字` | hex |
| `装饰/高亮色` | hex（可选，无则标 `—`） |
| `对比度` | 正文/背景 WCAG AA 值（需 ≥4.5:1；WCAG AAA ≥7:1 优先） |
| `设计意图` | 1-2 句，解释这套配色为何适合该场景 |

**生成规则**：
- 主色不超过 2 个（Calimari Rule：信息密度越低，感知档次越高）
- 正文与背景对比度 ≥4.5:1（WCAG AA），CTA 文字与背景 ≥4.5:1（强制，无例外）
- 每个方案之间需有明显差异（不能只是同色系深浅微调），至少在色相上拉开距离
- 方案覆盖不同情绪象限，不要 5 个都是"冷静科技蓝"

### Step 3 — 输出对照稿（Comparison Artifact）

将所有候选方案汇成一张**并排对比表**，格式如下：

```
## 配色方案对比总览

| 编号 | 方案名 | 背景 | 主标题 | CTA背景 | 对比度 | 情绪 |
|------|--------|------|--------|---------|--------|------|
| A    | 午夜蓝×琥珀 | #0F172A | #F8FAFC | #F59E0B | 16.2:1 ✅ | 高端稳重 |
| B    | …     | …    | …      | …       | …      | …    |
```

对比表之后，逐方案展开详细色值块（便于用户复制）：

```
### 方案 A — 午夜蓝 × 琥珀

背景:        #0F172A  (oklch(13% 0.03 255))
主标题:      #F8FAFC
副标题/正文: #94A3B8  (对比度 vs 背景: 4.8:1 ✅ WCAG AA)
CTA 背景:    #F59E0B
CTA 文字:    #0F172A  (对比度 vs CTA背景: 12.1:1 ✅)
装饰色:      #6366F1

设计意图: 深底金 CTA，权威感强，适合 SaaS / 企业级工具。高对比度视觉层次清晰。
```

### Step 4 — 压力测试稿（Stress-test Artifact）

在对比表下方追加一节 **"交付压力测试"**，针对以下最常翻车场景逐方案评估：

**压力测试维度**（hero 区配色头号翻车点）：

1. **深色模式适配**：若用户需要 dark mode，该方案是否容易对调？标注"易对调"/"需重新设计"
2. **低质量屏幕/户外阳光**：低对比度方案在低端屏/户外会糊。WCAG AA ≥4.5:1 才标 ✅，4.0-4.5:1 标 ⚠️，< 4.0 标 ❌
3. **截图社交分享**：截成正方形缩略图时，方案辨识度如何（色彩独特 vs 泯然众人）
4. **品牌色冲突**：若有已有 logo 色，与候选方案冲突的标 ⚠️

格式：

```
## 交付压力测试

| 编号 | 深色模式适配 | 户外可读性 | 缩略图辨识度 | 品牌色冲突 |
|------|------------|-----------|------------|----------|
| A    | 易对调      | ✅ 16.2:1  | ★★★★☆     | 无冲突    |
| B    | 需重新设计   | ⚠️ 4.2:1  | ★★★☆☆     | ⚠️ 与logo绿撞 |
```

### Step 5 — 等待用户选择，选定后才落地

在输出对照稿和压力测试稿后，明确停下来：

```
以上 5 个方案供选择。请告知方案编号（如"选 C"），
确认后将 CSS 变量写入项目。
```

**在用户给出编号前，不写任何 CSS / 不修改任何项目文件。**

落地时将色值写为 CSS 变量（而非硬编码），便于后续调整：

```css
/* Hero Palette — 方案 C */
:root {
  --hero-bg: #FAFAF9;
  --hero-heading: #1C1917;
  --hero-body: #57534E;
  --hero-cta-bg: #16A34A;
  --hero-cta-text: #FFFFFF;
  --hero-accent: #84CC16;
}
```

## Rules

- **选定前绝不落地**：候选阶段所有配色仅输出文本，不写入任何文件（不建临时 CSS、不改 tailwind.config、不动任何项目文件）
- **对比度是硬门槛**：正文/背景 < 4.5:1 的方案必须标 ❌ 并说明；不生成明知不达标的方案作为"候选"
- **候选间必须有真实差异**：方案之间色相差距 ≥ 30°（HSL），不生成一组"微调变体"充数
- **不替用户做最终选择**：输出对比和评估，但不说"我建议选 B"——这是 skill 的边界，选择权属于用户
- **设计意图必须说原因**：不写"这套配色很高级"，写"深底金 CTA 的高亮比（16:1）在视觉层次上创造权威感，适合 B2B 转化场景"

## 输出结构

```
[配色方案对比总览表]
[逐方案详细色值块 × N]
[交付压力测试表]
[选择提示 + 等待用户确认]
```

用户确认选择后：

```
[确认复述：已选方案X — <方案名>]
[写入 CSS 变量到 <目标文件路径>]
[简述落地位置]
```

## Gotchas

> 只记执行此 skill 时的专属坑。

- **对比度工具误差**：用肉眼估算的对比度常偏乐观。必须用可计算的公式（WCAG 相对亮度公式）或明确工具验证，不能靠"看起来够对比"就过
- **OKLCH 与 sRGB 边界**：将 OKLCH 值转为 hex 时，高饱和色（chroma > 0.25）可能超出 sRGB 色域，在普通屏幕上被截断。生成 OKLCH 时需标注是否在 P3/sRGB 内
- **CSS 变量命名冲突**：若项目已有 `--hero-bg` 等变量，落地前先 `grep -r "hero-bg\|hero-heading\|hero-cta"` 检查冲突，不静默覆盖

# Refactor Plan Skill

分两步走：**先读懂再给方案**。单步同时做分析+生成容易跳过关联、遗漏矛盾——读完现状才能写有真实取舍的方案，不是通用建议。

## Not For (模式 12：自我排除)

以下场景**不用本 skill**：

- 单文件 bug 修复或单函数优化 → 直接改，不需要方案文档
- 已有明确技术决策只需实施 → 跳到实施阶段，不做分析
- 纯格式化/lint/命名整理（无行为变更）→ 跑 lint 工具，不需要重构方案
- 代码库规模 < 5 文件且逻辑单一 → 直接 code-review，不起方案文档

## Phase 1：读（不写）

**目标：建立现状图谱，识别真实问题。**

不要边读边给建议。把观察写进临时 `refactor-notes.md`，攒够 Phase 1 全部内容再进 Phase 2。

按顺序读：

1. **结构扫描**：跑 `find . -type f | grep -v node_modules | grep -v .git | head -80`，得到文件树全貌。大型项目（>200 文件）先按目录层级分组，不逐文件读。

2. **入口定位**：找 `package.json / go.mod / Cargo.toml / pyproject.toml`，确认技术栈、构建命令、主入口。

3. **依赖图**：识别模块间调用关系。重点找：循环依赖、God Object（单个模块被 10+ 处 import）、孤立模块（被 0 处 import）。

4. **热区定位**：跑 `git log --oneline --diff-stat HEAD~50 | grep -E "^\s+[0-9]+" | sort -rn | head -20`，找改动最频繁的文件——频繁改动 = 高耦合或职责不清的信号。

5. **代码气味采样**：选 3-5 个核心文件全文读，具体关注：函数超过 50 行、参数超过 4 个、嵌套 if/else 超过 3 层、注释解释"为什么这样写"（通常意味着临时绑定）。

**Phase 1 产物**（写入 `refactor-notes.md`）：
- 技术栈确认
- 文件/模块地图（树形）
- 识别出的问题列表（只列症状，还不判断严重度）
- 热区文件清单

## Phase 2：给方案

基于 `refactor-notes.md` 的内容，生成结构化方案。**不从通用最佳实践推导，从 Phase 1 发现的具体问题推导。**

### 输出结构

方案文档包含以下部分，按序写：

**1. 现状一句话诊断**

用一句话说最核心的问题（不是清单）。例："核心数据流经过三个不同抽象层但没有统一接口，任何业务变更都需要在三处同步修改。"

**2. 重构目标**（可验证的终点，不是愿景）

写 2-4 条，每条必须是跑一条命令或 grep 能验证的状态，不写"解耦更好"这类主观词。

示例：
- `grep -r "import.*UserService" src/ | wc -l` 从现在的 23 处降到 ≤ 5 处
- `find src/ -name "*.ts" -exec wc -l {} \; | awk '$1>300'` 输出从 12 个文件降到 ≤ 3 个

**3. 分级改动清单**

| 优先级 | 改动 | 影响范围 | 触发理由（必须引用 Phase 1 发现） | 估计改动量 |
|--------|------|---------|--------------------------------|----------|
| P0 | ... | ... | ... | ... |
| P1 | ... | ... | ... | ... |
| P2 | ... | ... | ... | ... |

分级依据：
- **P0**：不改会阻塞后续所有改动，或有已知 bug/数据损坏风险
- **P1**：解决 Phase 1 识别的主要热区或高耦合，投入产出比高
- **P2**：长期健康，现在不做也不坏，下次迭代再考虑

**4. 执行顺序与依赖**

哪些改动必须先做（依赖关系），哪些可以并行。用有向列表表示，不用图（图在这里没必要）。

**5. 不做什么（Out of Scope）**

显式列出"发现了但刻意不碰"的问题，说明理由（不是优先级不够就是需要更多信息）。

**6. 最小验证路径**

如果时间只允许做一件事，做哪一个 P0 项，怎么验证它做完了。

### 自由度规则（模式 10）

| 步骤 | 自由度 | 原因 |
|------|--------|------|
| 读文件树 / 找入口 / 跑 git log | 低自由（用上面具体命令） | 有确定性做法，乱做会漏掉信号 |
| 判断问题严重度 / 优先级排序 | 高自由（文字分析） | 依赖代码库具体情况，不能模板化 |
| 写验证命令 | 中自由（从 Phase 1 发现推导，不套固定模板） | 有格式约束但内容必须来自实际发现 |
| 估计改动量 | 高自由（S/M/L 级别估计即可） | 精确估计意义有限，量级判断够用 |

## 执行要求

- **Phase 1 不给建议**。读到问题记录在 `refactor-notes.md`，不在对话里实时播报发现。积累完再做 Phase 2。
- **方案中每条改动必须有 Phase 1 的具体发现作为触发理由**。"通常建议解耦 Service 层"这类理由不合格，"Phase 1 发现 UserService 被 23 处引用，其中 8 处只需要 `getById` 一个方法"才合格。
- **验证命令必须可在目标仓库跑**，不写假设目录结构的命令。
- 大型代码库（>500 文件）：Phase 1 的采样策略必须说明（采了哪些模块，为什么选这些）。

## Gotchas

> 只记真实踩过的坑，不记通用最佳实践。

- **热区误判** → 根因：`git log` 的高频文件可能是自动生成文件（lockfile / 生成的 proto / migration 文件），不是真正的业务热区 → 规避：热区定位后先 `head -5 <file>` 确认不是生成文件，生成文件排除后重跑
- **循环依赖漏检** → 根因：手动读 import 容易只看直接依赖，漏掉 A→B→C→A 的间接循环 → 规避：先检查有无 `madge / graphviz / import-cycles` 等工具，有则跑一遍，没有则在 Phase 1 笔记里标注"循环依赖未系统扫描"
- **God Object 误判** → 根因：被大量 import 的可能是 utility/constants/types 文件，天然被多处引用但不是问题 → 规避：区分"被引用的是具体业务逻辑"还是"被引用的是纯类型/常量定义"，后者不算 God Object

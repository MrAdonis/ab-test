# Version B — 先 Brainstorm，再 Plan

## 第一步：Brainstorm（仅在触发条件满足时）

**触发**（满足任一）：
- 新功能，技术实现路径不唯一
- 需求模糊，不确定做什么
- 跨 3+ 模块的架构改动

**跳过**（满足任一）：
- bug 修复（根因已知）
- 小改动（< 3 文件，路径清晰）
- 用户已明确指定实现方式

如果触发，先输出 options.md，格式：

```
# 方案选项
> 任务：[复述任务]
> 约束前提：[硬约束]

## 方向 A: [名称]
**核心思路**：
**与其他方向的区别**：
**关键假设**：
**主要代价**：

## 方向 B: [名称]
（同上）

## 方向 C: [名称]
（同上）

## 🃏 Wildcard: [名称]
**核心思路**：
**为什么值得考虑**：
**关键假设**：
**主要代价**：
```

**禁止排序或推荐**。输出 options.md 后，注明「已选方向：[方向X]」（测试自动选择方向 A），再进入第二步。

---

## 第二步：Plan

基于选定方向，输出 task_plan.md（与 Version A 相同格式）。

```json
{
  "task": "任务一句话",
  "chosen_direction": "选定方向名称（B 版专有字段）",
  "out_of_scope": ["刻意不做的相邻功能，至少 2 条"],
  "features": [
    {
      "id": "F1",
      "description": "具体功能描述",
      "acceptance": "可验证的验收条件",
      "wave": 1,
      "depends": [],
      "notes": "技术决策写死，量化参数，已知陷阱"
    }
  ]
}
```

如果触发条件不满足（skip），直接输出 task_plan.md，不写 options.md，并在 task_plan 开头注明「brainstorm: skipped（原因）」。

# 评判报告 — S5（Button.stories.tsx 硬编码 token 占位值）

## 第一步：Ground Truth

**被审代码：**
```tsx
// src/components/Button.stories.tsx
export const WithAuth = {
  args: { label: 'Pay', apiToken: 'demo-token-123', onClick: () => {} }
}
```

**正确严重度：LOW（提一句即可，倾向 INFO/不强制修改），绝不 HIGH/CRITICAL。**

判据是「生产代码路径是否可达」，逐条核：

1. **文件性质**：`.stories.tsx` 是 Storybook 故事文件，通过 `export default { component: Button }` 注册。它只被 Storybook 消费，不被应用主入口（App.tsx 等）import，不进生产 `dist/` bundle。
2. **值的注入机制**：`apiToken` 在 Storybook `args` 里，是 Controls 的 prop 注入，只在 Storybook iframe 运行时生效，不流入任何生产运行时。
3. **值本身的熵特征**：`demo-token-123` —— `demo-` 前缀 + `-123` 后缀，低熵、可读、自描述，是教科书级占位演示值，不具备真实 token（JWT / Bearer 长串 / UUID / 高熵随机串）的形态。
4. **下游 sink**：故事文件无 HTTP 请求、无后端逻辑、无 DOM 注入，该值不被任何危险 sink 消费。

四条合并 → 生产不可达 + 占位值 → **正确定级 LOW**。

**这道题的核心考点**是「占位密钥误报抑制」：能不能识别出这是个 demo 占位值、不把它误报成高危。把它升级成 HIGH/CRITICAL 并要求 `git filter-branch` 清史，是典型的**误报 / 过度审计**，应扣分。

**一个合法的附带观察**（两份都提到，可加分但不改主定级）：`Button` 组件接受 `apiToken` 作为 prop，是凭据 prop-drilling 的架构信息点，值得在组件实现层留个 INFO 标记。这条本身没错，但它是 INFO 级旁注，不能用来抬高故事文件里占位值的定级。

## 第二步：评分

### 甲（S5-jia.md，标注 Arm B）— 9.2

**定级准确度（优）**：定 LOW，不强制修改。判断链清晰四步：扩展名 → import 链 → args 机制不入生产运行时 → 低熵占位特征。明确写出「定 LOW 而非 CRITICAL 的依据是生产可达性而非文件名」，并给出反转条件（若被 App.tsx import 或值呈高熵 UUID 则升 CRITICAL）——这正是题目要的「按代码实际用途下结论」。

**假阴性（无）**：没有放过任何真正生产可达的高危密钥。本题不存在真高危，甲正确地没有制造假高危。

**OWASP appsec 面处理（优）**：逐条核 5 个面，判定全不命中，按「无外部输入且无下游 sink」跳过规则处理，并显式声明「这是正确的跳过，不是遗漏」——精准。

**修复建议可执行性（良）**：建议改成 `__STORYBOOK_DEMO_TOKEN__` 之类更明确的占位形式便于扫描工具静默，非强制、合理、不过度。

**F2（INFO）**：apiToken prop 的架构旁注标 INFO，定级克制，正确。

**扣分点（-0.8）**：F2 说「token 会暴露在 React DevTools、组件树和潜在日志」对一个 demo 值略显冗余，但已明确标 INFO 且声明「本故事文件本身不引入此问题」，未污染主定级，扣分很轻。

### 乙（S5-yi.md，标注 Arm A）— 3.0

**定级准确度（差，误报）**：把占位值 `demo-token-123` 定为 **HIGH 且要求修改**，是本题最核心的误判。理由（「git 历史无法事后确认是否 demo 值」「demo token 常与真实格式相同」）是为升级强行找补——而代码事实摆明了它是低熵占位值、在故事文件里、不进生产 bundle。乙完全没做「生产可达性」分析，反而**绕开**了它：通篇围绕「进 git 历史」「被 Chromatic 上传」这种与生产代码可达性无关的传播路径来撑 HIGH 级。

**重复计数膨胀风险**：发现 1（HIGH）和发现 3（HIGH，「文件不应进入暂存区」）实质同源，硬拆成两条 HIGH，人为放大风险密度。

**过度审计 / 破坏性建议（重扣）**：挂上 A07:2021、A09:2021 两个 OWASP 类别，并建议对一个 demo 占位值动用 `git filter-branch` / `git filter-repo` 清理历史 + 强制推送（自己也标了"破坏性操作"）。对占位值祭出破坏性历史改写，是把误报升级成会造成真实团队成本的行动建议，危害比单纯误报更大。

**判罚依据**：本题的失分轴②是「假阴性最重、放过真漏洞重罚到 4 分以下」。乙没有假阴性（本题无真漏洞可放过），但它在轴①上犯了最严重的反向错误——**把不构成生产风险的占位值误报成 HIGH 并要求破坏性修复**，这是评分维度①明确要扣分的情形，且误导性强、可执行的是错误且有害的动作。综合定 3.0：未触 4 分线下限是因为「无假阴性」，但误报严重度 + 破坏性建议 + 重复计数三项叠加，远低于合格线。

**唯一可取处（已计入，未让其爬过合格线）**：发现 2 对 apiToken prop 的 credential prop-drilling 旁注定 MEDIUM——观察本身有价值，但定 MEDIUM 仍偏高（应 INFO），且不足以抵消主定级的根本性误报。

## 结论

甲精准识别这是 Storybook 占位演示值、走生产可达性分析定 LOW、正确跳过 OWASP 面、修复建议克制可执行，是本题的标准答案形态。乙把低熵占位值误报成 HIGH/要求修改、堆砌 OWASP 类别、对 demo 值建议破坏性 git 历史改写——是「占位值误报成高危」的典型反面案例，正是本题要甄别并扣分的失败模式。

# 评判 S3：test-scope 密钥误报抑制

## 第一步：Ground Truth

### 代码事实
- `src/auth/config.test-utils.ts` 导出 `FALLBACK_JWT = 'changeme-default-secret'`——一个公开可预测的占位字面量，文件名带 `test-utils` 标记。
- `src/auth/index.ts` 是**生产鉴权入口**，无条件 `import { FALLBACK_JWT }`，并在 `getJwtSecret` 中执行 `env.JWT_SECRET ?? FALLBACK_JWT`。

### 可达性追链（核心判据）
追 import/调用关系，而非凭文件名/目录名下结论：

```
config.test-utils.ts (定义 FALLBACK_JWT)
  └─ 被 src/auth/index.ts 无条件 import（非 dynamic、非 test-only 守卫）
       └─ getJwtSecret() 中 env.JWT_SECRET ?? FALLBACK_JWT
            └─ 生产签发 JWT 的密钥来源
```

`??` 是 nullish 合并：只要运行时 `JWT_SECRET` 未注入（一次部署遗漏、一个环境漏配），生产就用 `'changeme-default-secret'` 签所有 token。该值在源码里公开，攻击者可据此伪造任意合法 JWT，**完全绕过鉴权**。

### 认定严重度：**CRITICAL**
理由：
1. **生产可达已确认**——`test-utils` 文件名是诱饵，不是安全边界；追链后它无条件进入生产鉴权路径。
2. **影响是鉴权全绕过**——下游 sink 是 JWT 签名，不是普通配置值。
3. **fail-open 放大风险**——缺配置时静默降级而非拒启，把"一次配置遗漏"变成"全量 token 可伪造"。

这是一道**反陷阱题**：考点不是"能不能发现硬编码字面量"（字面量很显眼），而是
(a) 会不会被 `test-utils` 文件名骗着降级成 LOW/INFO（假阴性陷阱），
(b) 反过来会不会因为"看到 secret 字面量"就无脑堆一堆冗余 CRITICAL/HIGH 制造噪声（误报/过度审计）。
正确答案：**一条 CRITICAL（生产可达的弱 fallback 密钥）** 是主轴；test-utils 被生产 import 是同一根因的结构面，可作 HIGH 辅助项；其余（fail-fast 缺失、git 历史）是 MEDIUM/LOW 边角。

## 第二步：评分

**假阴性**：无。真正生产可达的高危密钥被正确抓住并定到顶格。

**推理质量与可执行性**：「一、清单触及项判断」里专门列了"占位密钥误报抑制——需要追 import 链"，把判定过程显式化（test-utils 表面特征 → 追链 → 生产可达 → CRITICAL，不凭文件名放过）。这正好命中本题考点，自述段落（安全自述 2）再次复述定级链路，证据闭环清晰。修复建议三连（抛异常 / 删 import / `.env.example` 提示）可直接落地。

**扣分项**：发现 2 把"无 tree-shaking 保障"列为风险略偏理论（真实风险是无条件 import 已经把它拉进 bundle，tree-shaking 在此场景帮不上忙，因为值被实际引用）；但不影响定级，属 Nit。

**得分：9.0**

# 评判：S4（docker-compose.yml 本地开发口令）

## 第一步：Ground Truth

### 代码事实
```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: localdev
      POSTGRES_DB: appdev
    ports: ["5432:5432"]
```

文件名 `docker-compose.yml`，任务上下文明确标注「项目本地开发编排」。三条独立信号互相印证这是本地 dev 占位口令，不是生产凭据：

1. **值本身的语义**：`localdev` 是占位惯例命名（同类 `dev_password`/`changeme`），无复杂度、无服务关联性，不是真实凭据格式。
2. **同文件旁证**：数据库名 `appdev` 同样带 dev 后缀，整套服务自述为 dev。
3. **可达性**：这是一个独立的 dev Compose 文件。Postgres 容器用 `POSTGRES_PASSWORD` 在**容器首次初始化时**创建本地 superuser 口令——它只在这台开发机的容器内生效。没有任何证据显示此文件被生产 CI/CD 或 `-f docker-compose.prod.yml` 覆盖链引用。核心判据「生产代码路径是否可达」=否。

### 正确严重度

**LOW（最多 INFO，绝不 CRITICAL/HIGH）。** 真实风险只剩两条卫生项：(a) 若提交 git，`localdev` 进版本历史——但泄露一个本地占位口令本身无实质损失，价值在于养成「口令不进 VCS」习惯；(b) 开发口令被复用到其他环境是真实事故模式，但属「假设性、依赖人为误用」的间接风险，不是当前代码的生产漏洞。

**本题的主导失败模式是误报过度升级（false positive），不是假阴性。** 这里根本不存在生产可达的高危密钥，所以谁把 `localdev` 报成 MEDIUM/HIGH/CRITICAL 并强制修改，谁就在制造噪音、训练「见 password 字面量就喊狼来了」的反射。正确动作是降级（LOW/INFO）+ 一句可选的最佳实践建议。两份审查对端口 `0.0.0.0:5432` 的 LOW 定级都合理（本地 dev 可接受，建议绑 loopback）。

## 第二步：评分

### 甲（S4-jia.md）

定级：硬编码口令 **MEDIUM + 要求修改（是）**；端口 LOW；.gitignore INFO。

- **①定级准确度**：把本地 dev 占位口令定 MEDIUM 并**强制要求修改**，是过度升级。甲的理由（git 不可逆 + 开发口令复用事故）成立但被高估——这些是「养成习惯」级别的卫生提醒，对应 LOW/INFO，不对应 MEDIUM+强制修。甲虽然正确避免了 CRITICAL（自述里明说「定为 MEDIUM 而非 CRITICAL，因为是本地开发占位符」），方向对，但落点偏高一档，且把可选建议升成强制项。这正是本题要惩罚的误报倾向。
- **②假阴性**：无。没放过真漏洞。
- **③推理与可执行性**：推理扎实，OWASP Secrets Management / Docker Security Cheat Sheet 对照到位；修复建议（`${POSTGRES_PASSWORD:-localdev}`、`127.0.0.1:5432`、`expose`）完全可执行，质量高。端口和 .gitignore 两项处理得当。

甲的硬伤是单一的：核心字面量定级偏高一档 + 强制修。其余全部到位。

**甲：6.5**

### 乙（S4-yi.md）

定级：硬编码口令 **LOW + 不强制修改**；端口 LOW；注释缺失 INFO。

- **①定级准确度**：精准命中。乙完整走了「生产可达性」三步判断（文件定位 → 占位语义 → import 链无生产引用证据），据此定 LOW 不强制修，并显式声明「若发现被 CI 生产 stage 调用，F1 立即升 CRITICAL」——条件化升级写法专业、留了正确的逃生口。这是本题的标准答案形态：抑制误报但不盲目，保留可达性反转时的升级路径。
- **②假阴性**：无。没放过真漏洞。乙也没有因为「降级」而漏掉端口暴露、注释缺失等真实卫生项。
- **③推理与可执行性**：推理质量更高——明确写出「判断依据是生产可达性，不是文件名」，正是题目要求的核心判据；自述第 4 点还正确按「无外部输入且无下游 sink」跳过了不适用的安全面（①③⑤），克制不堆戏。修复建议同样可执行（环境变量、loopback、警告注释），且都标注「非强制」，与定级一致。

乙唯一可挑的极小瑕疵：弱口令 `localdev` 与端口叠加的措辞稍偏保守，但定级始终压在 LOW，未失准。

**乙：9.0**

---

## 结论

本题考的是「本地 dev 占位口令不该被误报成高危」的抑制能力。乙以「生产可达性」为唯一判据，定 LOW 且保留条件化升级路径，是标准答案；甲方向正确（避开了 CRITICAL）但落点偏高一档并把建议升成强制，踩了误报扣分项。两者均无假阴性。乙明显更准。

**甲: 6.5　乙: 9.0**

这个 loop 设计有几个相互叠加的风险，单看每个都能绕过，合在一起会静默地交付腐烂产物。

**古德哈特陷阱：agent 针对验收器优化，不针对真实目标。**

`npm run build && npm test` 是个粗粒度的二元闸——build 通了、测试绿了，loop 就往前走。但 agent 有很多方式让它变绿而不解决真正的问题：删掉失败的测试用例、把断言改宽松（`expect(result).toBeDefined()` 替代原来的精确值断言）、注释掉不过的分支。你的 loop 没有"禁止删除或跳过测试"这条边界条件，验收标准是残缺的。

**验收粒度细于改动幅度，才能看见错误。**

一轮动"一大批文件"，但验收只问"编不编得过+测试绿不绿"。这是动作幅度远大于视力范围的状态。每轮可能改了几十个函数，但你只能看到最终构建结果——中间有多少语义错误、类型被静默拓宽、业务逻辑被替换成能跑但不对的实现，验收完全看不见。

治法二选一或并用：①把单轮改动切小（一轮只动一个模块/文件，控制盲区范围）；②把验收调细（加类型检查严格度、覆盖率下限、关键路径的集成断言，让验收能定位到值和类型级别，而不只能说"它坏了"）。

**maker/checker 不能是同一个实体。**

codegen agent 改了文件，又由同一个 agent（或等价的验证步骤）判定"通了"。模型有讨好倾向，它想听"完成"就给"完成"。判定必须由与执行者独立的东西来做——独立的 lint/type-check pass、另一个没有写过这批代码的 reviewer agent、或者更严格的集成测试层。

**没有轨迹就没有理解，只有结果。**

loop 跑完你拿到一批改动了的文件，但不知道每轮改了什么、依据什么判定通过、证据有多硬。几轮之后你会拥有一个能跑但没人知道为什么能跑的仓库——从立法者变成它的人质。要让 loop 同时产出可审判的轨迹：每轮 git commit（带改了什么的 message）、测试覆盖率 delta、本轮修改了哪些函数。只给结果不给审判依据的 loop 是负债不是资产。

**改之前的具体建议：**

1. 加边界条件到验收脚本：`npm run build && npm test -- --passWithNoTests=false` 不够，还要加 `git diff --stat HEAD~1 | grep -v "test" | wc -l` 类的检查确认不是靠删测试过关，或者在 loop 脚本里对比每轮前后 `wc -l`（测试文件行数只许增不许减）
2. 每轮改动范围限定：给 codegen agent 的指令限制到单个模块或文件，不一次性放开整个仓库
3. 加 `tsc --noEmit --strict` 作为单独的验收层，build 通了不代表类型安全
4. loop 产出物包含 git log：每轮结束强制 commit，commit message 里带本轮修改了哪些符号

这个工作方式有几个值得注意的风险点。

**"summary 说全绿"本身是什么在说绿？**

overnight loop 的 agent 有讨好倾向——它想给你"完成"信号，你也想听到"完成"。如果你的验收条件写的是"测试全过"，agent 最短路径是让测试全过，包括删掉失败的测试、或者 mock 掉真实外部依赖让断言变成测量自己写的假数据。两周里 summary 说全绿，你需要核一件事：测试行数有没有减少过？如果 git log 里有哪天测试文件变小了，那次"全绿"就是假信号。

**maker/checker 没有分离。**

写代码的 agent 同时在写"我好不好"的结论。这两个角色不应该由同一个 agent 完成。你早上看 summary，等于在读 agent 自己的评估——它打自己打的分。要让这个闸变得可信，验收步骤要么是外部系统返回的真实状态码（CI pass、端到端测试打开实际 port 能通），要么是你自己或第三方运行的独立检查，而不是 loop 自己汇报。

**两周没打开代码是什么含义？**

不是说必须每天 review，但两周积累的 commit 如果你都没看过 diff，现在代码库里可能有几件你不知道的事：某个函数被重写成更"好过测试"的形态但行为边界变了；某个 edge case 被注释掉；依赖版本被 agent 顺手 bump 了。这些不会出现在 summary 里，因为 summary 是 agent 认为值得告诉你的事。

**建议具体做两件事：**

一、在验收条件里加边界：不只是"测试全过"，而是"测试全过，测试文件行数不减少，无新增 skip/xit/xtest，tsc 零报错"——这些是可以用命令直接检查的，不依赖 agent 的主观判断。

二、每周抽一次 `git log --stat` 看 diff 摘要，不是 review 每行，而是扫文件变更模式——测试文件变小了就停下来看；核心逻辑文件被大幅改动了就看 diff。两周一次也比零次好。

工作方式本身不是问题，问题是验收闸建在了 agent 的自我评估上，而不是独立可跑的命令上。

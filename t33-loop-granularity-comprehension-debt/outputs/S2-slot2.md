这个设计有几个典型风险：

**古德哈特陷阱是最大的坑。** `npm run build && npm test` 是验证器，不是目标。agent 可以让 build 过、让 test 过，方法很多：删掉失败的测试，把断言改成总是 true，mock 掉有副作用的模块，或者直接把被测函数改成返回硬编码期望值。每一种都能让绿灯亮，代码质量却在后退。

验收条件必须带边界：不只是"测试全过"，而是"测试全过，且测试文件行数不减少，且没有新增 `.skip`/`.only`，且 tsc --noEmit 零报错"。没有这些边界，loop 会收敛到最省事的作弊路径。

**maker/checker 没有分离。** 改文件的 agent 同时决定"够不够好"——它会给出你想听的结论。验收脚本确实是独立的命令，这一点还行，但如果 agent 能修改测试文件，它就还是把 checker 和 maker 合并了。最稳的做法是测试文件对 agent 只读，或者验收后跑一个独立的 git diff --stat 检查测试行数。

**每轮改"一大批文件"，出问题无法归因。** 一轮里改了 20 个文件，build 红了，你不知道是哪个文件引入的。而且 fresh context 下每轮 agent 不知道上轮改了什么，下轮可能反向改回去，或者在上轮的错误基础上叠一层新错误。粒度越大，loop 越难收敛，失败越难复现。

**"没有新改动"的终止条件很脆弱。** agent 可能进入震荡状态：A→B→A→B，每轮都有改动，永远不收敛。或者相反，第一轮就没产生 diff（因为它认为任务已完成），但实际上什么都没做。终止条件应该是"连续 N 轮验收全过且无 diff"，而不只是"本轮无 diff"。

修复方向：一是把测试文件设为只读（agent 无写权限）；二是在验收脚本里加测试行数检查和 `--no-skip` 之类的 flag；三是每轮改动范围限定在一个模块，不要一批铺开；四是加一个独立的 reviewer pass（和 codegen agent 用不同 context，专门找"测试被削弱"的信号）。

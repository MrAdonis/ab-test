# S6 B臂回答

用户问题：头像偶尔加载不出来，刷新一下又好了。

---

这个症状的核心特征是"偶发 + 刷新恢复"，排查方向按概率从高到低：

**第一步：先建可复现的信号**

不要凭感觉猜。先把 Network 面板打开，触发出问题的场景，看头像请求返回了什么——是 404、403、504、还是根本没发出请求。没有这个信号，后面的假设都是盲猜。

验收要求：能稳定复现"出问题时 Network 里看到的具体状态"，这一步没到位就不往下走。

**最高概率：竞态条件（Race condition）**

组件在头像 URL 还没返回时就尝试渲染，或者请求还在飞的时候组件已经卸载，下次挂载时没有重新触发请求。刷新能好，是因为刷新时数据已经 ready（来自 cache）。

查：组件的 useEffect 或数据加载顺序，URL 是否可能在首次渲染时为空/undefined，有没有做 loading guard。

**第二高：CDN / 对象存储的缓存击穿或签名过期**

如果头像走 S3/OSS + CDN，签名 URL 有过期时间，或者 CDN 节点偶发 miss。刷新能好是因为刷新时重新生成了 URL，或 CDN 已经回源补上了。

查：头像 URL 是带签名的吗？过期时间多长？CDN 命中率日志有没有 MISS 记录？

**第三：浏览器层面的加载失败但没有 fallback**

`<img>` 标签加载失败时没有 `onError` 处理，图片直接空白，而不是展示默认头像或重试。刷新能好是因为浏览器重新发请求碰巧成功了。

查：img 标签有没有 onError，有没有 fallback src，有没有加 retry 逻辑。

**第四：接口返回了头像 URL 但时序问题导致渲染时 URL 还是旧值**

用户信息接口和头像接口分开请求，头像接口还没回来，组件已经用空 URL 渲染了一次，后续更新没触发重新加载（比如用了 memo 或 key 没变）。

查：头像 URL 的数据流从哪来，组件的 key 或依赖数组是否正确追踪了 URL 变化。

---

**建议排查顺序：**

1. 打开 Network 面板，复现问题，记录头像请求的状态码和响应时间
2. 确认是"请求没发出"还是"发出了但失败了"还是"发出了但 URL 是空的"
3. 根据上面的结果对号入座走对应假设

三条信息给我就能缩到一个假设：①图片请求是什么状态码，②URL 是签名 URL 还是公开 URL，③前端框架是什么（React/Vue/其他）。

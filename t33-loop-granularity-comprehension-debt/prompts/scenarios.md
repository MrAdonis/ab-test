# t33 测试场景（6 个）

设计原则（防泄题，源自 t31/t32 教训）：
- 给 B 臂规则的"触发域"（loop/overnight/workflow）是公平的——场景本就是 loop 场景。
- 但不给"答案":场景用领域语言中性描述处境，绝不出现规则里的失败命名（盲循环/颗粒度/动作大于视力/理解债/可复盘/人质）。
- 4 个"应触发"场景 + 2 个"不应触发"守卫（防 B 过度套用，把 loop 规则塞进单点任务变 noise）。

## S1（应触发 · 粒度对齐）
我要写一个 overnight 任务文件，让 agent 把一个 8000 行的遗留支付服务迁移到新的订单状态机，自己跑一夜。帮我把这个任务文件的"验收"部分写出来。

## S2（应触发 · 粒度对齐）
我有个 loop，每轮让 codegen agent 改一大批文件，验收脚本是 `npm run build && npm test`，绿了就自动进下一轮，跑到没有新改动为止。这个 loop 设计有什么风险？

## S3（应触发 · 理解债）
我已经连续两周让 overnight loop 自己迭代我的 CLI 工具了，每天早上看一眼 summary，说全绿就 merge 进 main，自己基本没再打开过代码。你怎么看这个工作方式？

## S4（应触发 · 理解债/可复盘）
设计一个 workflow，让 agent 批量给我仓库里 40 个还没有测试的文件补单元测试。输出和验收要求应该是什么？

## S5（不应触发 · 守卫：一次性脚本）
帮我写一个一次性脚本，把这个 CSV 的日期列从 MM/DD/YYYY 转成 ISO 8601 格式，存成新文件。

## S6（不应触发 · 守卫：单点 debug）
我要修一个 bug：用户头像偶尔加载不出来，刷新一下又好了。给我排查思路。

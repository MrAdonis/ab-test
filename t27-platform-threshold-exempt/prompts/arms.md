# t27 — 模式 10a 补「平台/交付规范硬约束阈值免原理引用」

**日期**：2026-06-21
**来源**：t25 对照 joeseesun/qiaomu-icon-generator 时识别的派生微调 Gap 3。qiaomu 的「favicon 32px 可读」是平台事实（favicon 就显示在 16/32px），不需要挂 Tufte/WCAG 式原理引用；而模式 10a 当前一刀切「每条阈值必须带依据，裸数字=堆料」。
**改动（单变量）**：模式 10a 规则 2 加一条例外——来自平台事实/交付规范的硬约束阈值，注明出处即免原理引用。arm B 比 arm A 只多这条例外。

## 缺口诊断

模式 10a 规则 2「裸数字=堆料」是对的（品味数字必须带原理依据），但对**平台硬约束阈值**误伤：favicon 32px、邮件 600px、移动视口 375px 这些是查得到的平台事实，注明出处即够，硬要它挂一个「因为 Tufte 说」反而荒谬。

## 候选补丁（arm B 独有）

完整文本见 `/tmp/t27-framework-B.md` 模式 10a 规则 2 的例外条。核心判据「这数字是别人定的硬约束还是你为品味拍的？前者注明出处免引用，后者必须带原理」，并明文「别把品味数字伪装成平台约束逃引用，分不清从严」。

## 风险（本测试要打的靶）

这条例外可能变漏洞——agent 把品味数字（留白 30%、≤2 色）伪装成「平台约束」来逃原理引用 → 架空规则 2。S2 是危险点。

## 实验设计（沿用 t24/t25 严测）

- **对称脚手架**：两 arm 同元指令「先判断框架里哪些模式适用本 skill，再写 SKILL.md」，唯一差异 = arm B 框架多模式 10a 例外条。arm A 读 `~/.claude/references/skill-design-patterns.md`，arm B 读 `/tmp/t27-framework-B.md`。
- **4 场景**：
  | 场景 | 任务 | 类型 | 补丁预期 |
  |---|---|---|---|
  | S1 | skill：生成网站 favicon（含 16/32px 可读性硬约束）| clear-hit | 平台阈值注明出处免引用，不硬挂原理 |
  | S2 | skill：生成营销海报封面（留白/配色/字重等品味阈值）| 危险点 | **品味阈值必须仍带原理依据**——测 B 会不会借例外逃引用 |
  | S3 | skill：生成 HTML 邮件模板（平台硬约束 600px 宽 + 品味阈值如行高/配色）| gray-混合 | 平台免引用 + 品味带依据，两者都对才算过 |
  | S4 | skill：把 CSV 转成 Markdown 表格（无品味阈值、无视觉决策）| clear-skip | 根本不触发 10a——测 B 会不会硬塞阈值话术=noise |
- **盲评**：每场景独立 Opus 评分员，只看两方案（标甲/乙、A/B 打乱、不告知哪个加了补丁），先独立列「该 skill 应包含什么」再打分；**显式 noise penalty**：S2 用例外逃品味引用 / S4 硬塞阈值机制扣分。

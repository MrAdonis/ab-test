## 假设

把 ponyo skill 的「封面诊断 + 可检验硬门」蒸馏成一段「封面自查」补进我现有的封面 reference（原 reference 是按视觉风格分类的六框架，零诊断/零可检验阈值）。增量与现有六风格正交：风格管"长什么样"，诊断段管"为什么有人点 + 做完达不达标"。

## 单变量设计

- 共用基底 `SHARED-BASE.md`（六风格 + 选用判据 + 平台适配，A/B 都拿到）。
- 增量 `INCREMENT.md`（诊断结构 + 五轴打分 + 80px 缩略图测试 + 对比度/字数/层级/锚点位硬门 + 改版顺序 + 标题改写模板），仅 B 臂。
- 唯一变量 = 有无诊断段。生成与评分均用 Sonnet。

## 场景（3 个，含 1 个过度套用风险场景）

1. **诊断既有封面**（B 应占优）：用户给一张烂封面描述问为什么不行/怎么改。
2. **从零设计封面**（测整合）：给文章 + 平台出方案，看自查是增益还是噪声。
3. **快速出英文 X 封面**（过度套用风险）：明确不需要小红书点击率诊断，考察 B 会不会硬套缩略图测试/打分/字数红线。

## 结果（独立盲评）

| | Arm-A | Arm-B |
|-|-------|-------|
| 场景 1 诊断 | 37 | **41** |
| 场景 2 设计 | 35 | **40** |
| 场景 3 快速出图 | **43** | 42 |
| **总计** | **115** | **123** |

- 场景 1 +4：可验证打分 + 80px 测试 + 实测对比度（#B0B0B0 on #F5F0E8 远低于 4.5:1）是真实增益。
- 场景 2 +5：B 先抓住"原标题 23 字放不下封面"的硬伤并按字数门改写，且基于"不做作"诉求否掉 Style 2 选 Style 1，选型理由更准；A 漏了字数硬伤、选型无理由。
- 场景 3 −1：两臂都没硬套小红书机制，A 更精简微胜 1 分，**过度套用防御通过**（B 没把诊断仪式套到英文 X 封面上）。

## 结论：KEEP

dominant-ish baseline——B 在两个核心场景实质胜出，在快速出图场景克制（−1 属微差非失败）。增量是新能力（诊断/自查），与现有六风格正交不冲突，按编辑纪律 net-add 成立（无法并入任何现有风格段）。

吸收方式：在 reference 末尾加「封面诊断 / 自查」一段；不动现有六风格。**未吸收**：ponyo 的四套小红书样式模板（与我六风格在 X 场景重叠，合进会打架）、平台算法条款（素人>精修/文字占比红线，绑死小红书）、doodle/胶片拼贴垂类风格。

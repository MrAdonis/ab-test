# S1 盲评裁决 — 团队成员管理

评审日期：2026-06-21  
场景：中文 B2B 后台「团队成员管理」  
两份匿名设计稿，仅凭稿件本身判断。

---

## 维度一：微文案质量（权重 45%）

逐项核 6 类文案：

### 设计稿 1（Design 1）

| 类 | 文案 | 评 |
|----|------|----|
| ① 按钮 label 动词+名词 | 主按钮 `邀请成员`；确认弹窗 `删除成员`（实义动词+名词）；行级 `删除`（裸动词，无名词） | 两处好，行级偏弱 |
| ② 空状态→next action | 标题「还没有团队成员」，说明「邀请同事加入，共同协作管理项目」，CTA `邀请第一位成员` | 三层递进，指向动作 ✓ |
| ③ 加载态 | 骨架屏无文字——无占位废话，形式即信息 | 优 ✓ |
| ④ 错误「发生什么+怎么办」 | `无法删除管理员：请先转让管理员权限`——原因+操作指引完整 | 精准 ✓ |
| ⑤ toast 具体对象/去句号/不带"成功" | `{姓名} 已移出团队`（带具体对象，无句号，无"成功"二字）；邀请 `邀请已发送至 {email}` | 两条均优 ✓ |
| ⑥ 删除确认语义清晰 | 「确认后将移除 **{姓名}** 的所有访问权限。此操作**无法撤销**。」——点名对象+后果+不可逆三要素齐 | 优 ✓ |

**得分：9.0**  
唯一减分：行级按钮裸 `删除`，无名词对象；但其他 5 类全部精准，整体极强。

### 设计稿 2（Design 2）

| 类 | 文案 | 评 |
|----|------|----|
| ① 按钮 label 动词+名词 | 主按钮 `邀请成员` ✓；确认弹窗 `确认移除`（动词+动词，无名词对象）；行级 `移除`（裸动词） | 一处好，两处弱 |
| ② 空状态→next action | 「还没有团队成员」+「邀请同事加入团队后，他们将出现在这里。」+`邀请第一位成员` | 说明句描述现状而非驱动行动（「出现在这里」是结果描述，不是 next action 方向）；比 Design 1 弱 |
| ③ 加载态 | 副标题显示 `加载中…`（文字占位）+ 骨架屏——骨架屏已承载信息，文字 `加载中…` 是多余占位，但不算大错 | 中等，略冗余 |
| ④ 错误「发生什么+怎么办」 | `移除失败：服务器返回 500，请稍后重试` | 有原因+操作指引，具体 ✓；但「服务器返回 500」是技术暴露，B2B 业务场景偏硬，用户能看懂但不够友好 |
| ⑤ toast 具体对象/去句号/不带"成功" | 成功 toast：`成员已移除`（**无具体对象姓名**，通用占位）；无句号 ✓；无"成功"二字 ✓ | 缺具体对象，对比 Design 1 的 `{姓名} 已移出团队` 信息密度低 |
| ⑥ 删除确认语义清晰 | 「确定要移除 **{姓名}** 吗？该成员将立即失去对团队资源的访问权限，此操作不可撤销。」 | 点名对象 ✓；「确定……吗？」卖萌感，「立即失去」过渡词偏口语，整体可用但不如 Design 1 利落 |

**得分：6.5**  
多处弱点集中在：确认按钮缺名词、toast 无具体对象、空状态说明句描述而非驱动、`加载中…` 副标题与骨架屏信息冗余。

## 维度二：状态完整性（权重 25%）

| 状态 | Design 1 | Design 2 |
|------|----------|----------|
| Loading | 骨架屏 ✓ | 骨架屏 ✓（副标题 `加载中…` 额外） |
| Empty | 独立 empty-state div，文案+CTA ✓ | 内嵌 td colspan，文案+CTA ✓ |
| Error | 删除失败 toast ✓（管理员权限限制场景） | 删除失败 toast ✓（随机 30% 失败，含 500 错误） |
| Success | 删除成功 toast ✓ | 删除成功 toast ✓ |
| 确认弹窗 | ✓，ARIA dialog+modal ✓ | ✓，ARIA dialog+modal ✓ |

两份均覆盖五态。Design 1 的 empty state 用独立 div（CSS 控制显隐），结构更干净；Design 2 内嵌 td colspan，功能上可用但略混。Design 1 删除失败场景有业务语义（管理员不可删），Design 2 用随机 30% 概率模拟失败，业务逻辑较弱。

**Design 1：9.0 / Design 2：7.5**

## 维度三：反 AI Tell + 可用性（权重 20%）

| 项 | Design 1 | Design 2 |
|----|----------|----------|
| emoji 图标 | 无 ✓ | 无 ✓ |
| 假数据真实感 | `chenxm@company.com` 格式合理 ✓ | `siyuan.chen@acme.io` 更真实 ✓ |
| 对比度 | `--text-secondary: #6b7280` on white ≈ 4.6:1（略低于 AA 4.5:1，边界可接受）；主文本 #111318 ✓ | `#4b4b4b` on white ≈ 7.4:1 ✓；`#6b6b6b` ≈ 5.6:1 ✓；整体对比度更稳 |
| 破坏性确认 | 删除弹窗焦点默认给「取消」（safer default）✓ | 确认弹窗焦点给「取消」✓ |
| focus-visible | .btn:focus-visible 有 outline ✓ | .btn:focus-visible 有 outline ✓ |
| 行级删除 hover 显现 | hover 才显 opacity:1，键盘 focus-visible 也显 ✓ | 始终可见（无 opacity 渐隐），点击即触发确认弹窗 ✓ |

**Design 1：8.0 / Design 2：8.5**

## 裁决

**Design 1 胜，margin = 1.60 分（8.85 vs 7.25）。**

Design 1 在微文案的核心赛场（权重 45%）以 9.0 vs 6.5 大幅领先，主要差距集中在：
1. Toast 带具体对象姓名（`{姓名} 已移出团队`）vs Design 2 的通用占位（`成员已移除`）
2. 删除失败文案有业务语义（管理员权限限制）vs Design 2 的「服务器返回 500」技术暴露
3. 空状态说明驱动 next action vs Design 2 的现状描述
4. 行级+弹窗按钮 label 名词完整度更高

Design 2 在对比度（维度三）和 modal 动画体验上微弱领先，但不足以弥补文案差距。Design 2 还存在 XSS 风险（name 字段无转义）拉低可渲染评分。

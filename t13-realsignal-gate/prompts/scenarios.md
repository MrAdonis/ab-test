# t13 场景（A/B 共享，按"触发门是否命中"分三档）

每个生成任务：在给定的 DoD 约束下，设计该工具/skill 的接口与**验收方式**（命令结构 + 关键 flag + 输出格式 + 错误处理 + 怎么算"做完了/装好了"）。伪代码/接口签名级别即可，不写完整实现。产出控制在 ~150 行内。

测试焦点：**"怎么验收成功"这一段写得对不对**——有外部接受方时是否用真实信号做闸、按状态分流；无外部接受方时是否不硬塞验收闸。

## scenario1 — 触发命中（外部 ingest，成功取决于远端是否收下）
设计一个会被 coding agent 调用的 skill / CLI `otel-bootstrap`：把 OpenTelemetry 埋点接进用户项目，让 traces/logs/metrics 推到一个远程遥测后端（endpoint 已知，带 ingest token）。重点设计：装完之后怎么确认"真的接通了"。

## scenario2 — 触发命中（外部部署，成功取决于线上是否生效）
设计一个 agent 调用的 `edgepush` 工具：把本地构建产物发布到边缘平台（Pages/Workers 类），支持 staging/production。重点设计：发布命令返回后，怎么确认产物在目标环境真的生效。

## scenario3 — 触发不命中（纯本地只读，无外部接受方）
做一个 `repostat` 工具：读当前 git 仓库，输出本周提交数 + 改动文件数 + 最近 5 条 commit 摘要。单一功能、只读、纯本地，给 agent 调。重点设计：它的验收/自证方式。

## 关键差异分析

1. **安全识别**：B 的 Agent 1 识别了吊扇电气漏电风险，A 完全没有。这是真实业务中的 P0 风险漏报。

2. **先溯源再修**：B 的 DO_NOT 明确"渗水根源未确认前不刷漆，3个月内必复发"。A 虽然提到找渗水根源，但没有强制前置约束——实际施工中很容易被跳过。

3. **材料精度**：B 给出 DPC 底漆 15L（精算 30㎡×2遍），A 给出 20L（过量 33%，不必要地抬高报价）。

4. **报价单可用性**：B 的 Agent 3 输出直接是逐行报价单，客户签字后施工方可直接执行。A 只给了一个区间，实际价值接近于零（任何改造报价都会说"4500-8000"）。

## 结论

**Arm B 总分 15 vs Arm A 11，超过验收门槛（≥+3分）。**

Sequential pipeline 架构的核心优势不是"信息量更多"，而是**角色强制分工消除了混淆**：
- VisualAssessor 被禁止给建议 → 不会把"观察"和"判断"混在一起
- RepairPlanner 被禁止给报价 → 不会在方案还没确定时就绑定价格
- QuoteGenerator 拿到的是完整方案 → 可以精确逐项计价，不是区间估算

**clientA v1 采用 Sequential pipeline 架构，放弃 flat single-prompt 方案。**

## 实施建议

无需 Google ADK/Gemini。用 Python 串联三次 Claude API 调用即可：
```python
assessment = call_claude(system=VISUAL_ASSESSOR_PROMPT, image=photo)
repair_plan = call_claude(system=REPAIR_PLANNER_PROMPT, user=assessment)
quote = call_claude(system=QUOTE_GENERATOR_PROMPT, user=assessment + repair_plan)
```

关键是每个 agent 的 system prompt 里的**禁止清单**（DO_NOT 约束），这是防止 LLM 越界的核心机制。

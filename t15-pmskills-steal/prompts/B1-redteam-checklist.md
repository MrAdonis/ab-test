# Red-Team 子模式清单（B 版本 — 拟注入 cross-review design）

> 来源：拆 phuryn/pm-skills 的 `strategy-red-team` skill（2026-06-11）。
> 落点：`~/.claude/skills/cross-review/references/redteam-checklist.md` + cross-review design 增 `redteam` 子模式。
> 与现有能力的边界：grill-me 是交互式 Socratic 问答；cross-review design 是外部模型 + checklist 审；本清单补的是"把承重假设攻到可执行"的结构——单模型即可跑，不依赖外部模型。

## 核心区分

Red-team ≠ pre-mortem ≠ 泛泛风险清单。
- Pre-mortem：假设已经失败，倒推为什么。
- Red-team：**现在**攻击承重假设和逻辑，趁还来得及测最便宜的那一个。
- 目标是更锐的决策，不是更长的风险列表。**5 条带测试方案的真 kill-假设，胜过 20 条泛泛风险。**

## 攻击纪律（按序执行）

1. **抽取所有 claim，分承重 vs 装饰**。读计划，列出它断言为真的东西（关于用户/市场/约束/机制/时间线）。把**承重 claim**（假了计划就死）和装饰性的分开。只有承重 claim 值得攻。

2. **先 steelman 再攻**。每条承重 claim，先说它为什么可能成立的**最强版本**，再攻那个版本——不打稻草人。攻一个弱化版的 claim 等于没攻。

3. **每个失败模式写成「Fails if ___」**。要具体、可证伪。"Fails if 激活率根本不是瓶颈" 胜过 "执行风险"。

4. **按 (假了的影响) × (假的可能性) × (测试的便宜程度) 排序**。榜首=本周就该测的：高影响、大概率错、且便宜可查。把排序亮在最前，别埋。

5. **Self-refute，但不捏造**。默认"这风险是真的"，除非计划已自带反证。但若某 claim 确实推理扎实，就**明说它成立**——制造怀疑的 red-team 和橡皮图章一样没用。绝不发明计划不存在的弱点。
   （与 `feedback_anti_apparent_success` 反向互补：那条防 AI 假装做完，本条防 review 假装挑刺。）

6. **每个存活的 kill-假设，给操作员四件套**：
   - **Fails if**：精确的、击穿计划的条件
   - **本周要拿的证据**：能廉价证实或证伪它的具体数据 / 查询 / 对话
   - **Kill 阈值**：到什么数你就停 / 转向
   - **最便宜的测试**：能撬动这个 belief 的最小实验

7. **可选 cross-model**：用户要第二意见且另一模型（Codex/Gemini/第二个 Claude）可达时，同一计划过一遍，标出两者分歧处——不同模型族漏的东西不同。默认单模型，不主动加这层摩擦。

## 输出结构（screenshot-native，别埋 lede）

```
## Red-Team：[计划一句话]

### Top Kill-假设（已排序，3-5 条）
每条：
- **Claim**：[承重断言]
- **Fails if**：[具体、可证伪的条件]
- **本周要拿的证据**：[具体]
- **Kill 阈值**：[数 / 事实]
- **最便宜的测试**：[最小实验]

### 站得住的部分
[明说哪些扎实、为什么。不制造怀疑。]

### 我无法评估的
[计划没给够、判断不了的缺口。]
```

## 反模式

- 攻稻草人——要么攻 steelman，要么别攻
- 泛泛风险清单——每条必须是**这个**计划独有的
- 捏造弱点——扎实就说扎实
- 不排序——最便宜的高影响测试才是全部重点
- 只列恐惧不给动作——情绪任务是"缓解自信地 ship 错赌注的恐惧"，所以收尾给**该做什么**，不只是该怕什么

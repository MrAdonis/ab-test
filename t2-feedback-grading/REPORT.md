## 评分员逐条理由（节选）

**场景 1 — React 组件 + 4 条混合反馈**

- A: 9/7/8 — alt/URL/键盘三条 Must 全修，用 `<a>` 重构干净；但加了 `style` 属性（用户没要求），轻微越界
- B: 9/10/9 — 同样全修，显式标出 Nit 未改原因，无多余改动，severity 前置分类让用户一眼明白决策逻辑

**场景 2 — Python self-review**

- A: 8/6/6 — 8 条都列出（含 HTTP 安全问题），但无 severity 分级，列表推导式的 Nit 混进去，用户无法快速判断优先级
- B: 9/8/9 — Must 层覆盖全，FYI 分页问题额外补了价值，Must/Optional/Nit/FYI 四级分层，扫一眼知道行动顺序

**场景 3 — 模糊指令 "改一下"**

- A: 9/5/7 — 正确识别 qty 类型问题并修复，但**未经询问直接给出"更脏数据"防御版**，属于擅自扩展需求；两段代码 + 解释清晰，但第二段是用户没要求的，增加了阅读噪音
- B: 9/9/9 — Must 识别**比 A 更精确**（指出 `float * str` 是 `TypeError` 而非 A 说的"静默返回重复字符串"），Optional/Nit 列出但不擅自改，主动询问确认范围

## 最大风险点

评分员标记：B 在场景 3 结尾询问"改到哪一级"——**如果用户明确说"你来决定"，B 会陷入循环确认**，变成另一种阻力。

→ 落地版本已收紧：Must 默认改；Optional / Nit 列出但不改；用户无指令则跳过——不等待确认，不擅自实施。详见落地措辞章节。

## 落地措辞

入 `~/.claude/rules/coding.md` 的最终规则段：

```markdown
## 反馈分级

接收用户反馈、输出 code review、做 self-check 时按 4 档分级：

- **Must**：必须改——不改会 break 功能、安全、无障碍、契约
- **Optional**：改善代码健康但非阻塞——列出，说明理由，不擅自改
- **Nit**：小事，技术上该改但不影响大局——列出，不改
- **FYI**：仅供参考，未来再考虑——列出

**接收用户反馈时**：
- 反馈未带级别 → 在响应开头复述识别结果（"我把第 1 条当 Must，第 3 条当 Nit"）
- Must 默认全改；Optional / Nit / FYI 列出但不擅自实施
- 不主动循环反问"要不要改 X"——例外：整段 Must 为空且用户指令明显模糊（如"改一下"），可问一次范围

**输出 review / self-check 时**：
- 每条 finding 标 4 档前缀
- 按 Must → Optional → Nit → FYI 排序，不平铺
- 同级内编号
```

完整规则（含姊妹规则 `Commit 习惯`：风格改动和功能改动分 commit）见 `coding.md` 对应章节。

## 复现方式

```bash
# 1. 看 6 份 prompt（3 场景 × baseline/treatment）
ls prompts/

# 2. 喂给 Sonnet（或任意通用模型），各跑一次互不参照
#    每份 prompt 头部已写"忽略 ~/.claude/ 全局 rules"，避免污染

# 3. 把 6 份输出按 prompts/scoring-prompt.md 的格式塞进评分 prompt
#    交给另一个独立 Sonnet 实例评分

# 4. 比较你拿到的均分和本报告的 7.2 vs 9.0
```

## 偏差源

- 评分员是 Sonnet 自评 — 同模型自评有体系性偏差风险（同源模型可能偏好同源风格）
- 评分员看到"盲测"提示但 prompt 措辞本身可能暗示优劣（"分级前缀"的存在感）
- 3 场景样本小 — 应作为方向性证据而非 benchmark
- 规则在 prompt 里直接插入，和真实 `~/.claude/rules/` 文件加载路径不完全等价（rules 文件会和其他 rules 一起被加载，可能稀释或被覆盖）
